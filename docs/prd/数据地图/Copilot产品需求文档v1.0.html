1. 背景与目标

1.1 项目背景

数据地图首页已完成 UI 重构（全局搜索 + 最近访问 + 资产大盘），但核心的 AI 智能助手（Copilot）模块尚未在需求文档中完整定义其交互细节和产品逻辑。

Copilot 是数据地图"从人找表到表找人"智能化升级的核心载体，承担着自然语言找表、个性化推荐、反馈闭环三大职责。本文档将 Copilot 作为独立模块，详细定义其全部功能、交互流程和技术契约，供研发团队评审和开发。

1.2 产品目标

序号 目标 说明 衡量指标
1 降低找表门槛 用户无需知道表名/库名，通过自然语言描述业务需求即可获得推荐 搜索转化率 > 60%
2 主动推荐（表找人） 基于用户画像（部门、角色、负责表血缘），打开即推荐高相关表 推荐点击率 > 30%
3 建立反馈闭环 点赞/点踩 + 结构化反馈，为推荐算法持续迭代提供数据 反馈率 > 15%
4 多场景覆盖 支持侧边栏快捷问答 + 全屏沉浸式对话两种模式 全屏使用占比 > 20%

1.3 功能边界（本期范围）

本期（v1.0）包含：
Copilot 侧边栏 / 全屏模式、欢迎区与智能推荐、四大意图找表、对话交互（含打字机效果）、点赞/点踩反馈、会话管理（新建/切换/重命名/删除）、用户消息编辑（覆盖重写/仅回退）
本期不包含：
血缘问答与图高亮、SQL 自动生成、字段级业务翻译、真实大模型接口对接（本期使用 Mock）、后台推荐策略配置

2. 目标用户与核心场景

2.1 目标用户

用户角色 典型任务 使用频率
数据开发工程师 按表名/字段名精准查找、查看血缘上下游 高频（日均 5-10 次）
数据分析师 按业务场景模糊查找（如"用户留存分析需要哪些表"） 中频（日均 2-5 次）
业务人员 / PM 不懂技术术语，用自然语言描述数据需求 低频（周均 3-5 次）
新入职员工 了解公司有哪些数据资产，快速上手 集中（入职首周高频）

2.2 核心用户场景

场景一：打开即推荐（表找人）

角色：数据开发工程师张三，交易数据组

行为：每天打开数据地图，点击顶部 Copilot 按钮

预期：Copilot 自动问候 "Hi 张三，你好！作为交易数据组的数据开发工程师..."，并推荐 3-5 张与其部门/负责表血缘相关的高频表卡片

价值：零输入即可获得个性化推荐，缩短每日找表路径

场景二：按业务用途模糊找表（人找表）

角色：数据分析师，不知道具体表名

行为：点击意图卡片"按业务用途找表"，输入 "我想分析最近一个月的渠道转化漏斗"

预期：Copilot 返回解释文本 + 推荐表卡片列表，用户点击卡片直接跳转到表详情页

价值：自然语言描述业务需求即可定位数据表，无需了解库表命名规则

场景三：按关键词 / 项目 / 专题范围精准找表

角色：数据开发工程师，明确知道要搜索的范围

行为：点击"指定项目范围找表"，输入 "风控项目里有哪些用户画像表"

预期：Copilot 在指定范围内检索，返回匹配结果

价值：缩小检索范围，提升精准度

场景四：对推荐不满意，提交反馈

角色：任意用户

行为：AI 推荐的表不准确，用户点击"踩"，在内联反馈卡片中勾选"找不到想要的表"并补充说明

预期：反馈卡片在消息下方平滑展开（不弹窗），提交后显示"感谢反馈"

价值：结构化反馈数据驱动推荐算法迭代

场景五：全屏沉浸式多轮对话

角色：数据分析师，需要深度探索

行为：在侧边栏点击"全屏"按钮，进入独立 Copilot 页面，左侧查看历史会话，右侧持续对话

预期：全屏模式下欢迎区更大、对话空间更宽敞，支持会话切换和管理

价值：复杂分析场景下的沉浸式体验

3.2.4 推荐列表产品逻辑

推荐列表是 Copilot "表找人"能力的核心，需要在用户零输入的情况下，精准推荐 3~5 张高相关性数据表。以下定义完整的推荐策略和排序规则。

Step 1：候选集召回（多路召回）

系统从以下 4 个维度并行召回候选表，每路最多返回 20 张表：

序号 召回路 逻辑说明 权重系数
1 血缘关联召回 获取当前用户负责的所有表（ownedTables）→ 查找这些表的上游 1~2 跳和下游 1~2 跳血缘表 → 去除用户自身负责的表
核心逻辑：你负责的表的上下游，大概率是你日常需要关注的表 		0.35
2 部门热表召回 获取当前用户所在部门（department）→ 统计该部门所有成员在过去 30 天内查询频次 Top 20 的表 → 去除用户自身负责的表
核心逻辑：同部门同事高频查询的表，对你也可能有价值 		0.25
3 个人行为召回 获取当前用户过去 30 天内的搜索关键词 + 浏览记录 + 收藏列表 → 基于关键词匹配 + 协同过滤，召回相关表
核心逻辑：你最近在找什么，系统帮你找更多相关的 		0.25
4 角色画像召回 基于用户角色（role）匹配预定义的角色-表域映射规则：
- 数据开发 → 偏好 DWD/DWA 层表
- 数据分析 → 偏好 DM/ADS 层表
- 业务人员 → 偏好 ADS 报表层
→ 在对应分层中按全局热度排序召回
核心逻辑：兜底策略，确保新用户也有合理推荐 		0.15

Step 2：去重与合并

  • 以 fullyQualifiedName 为唯一键，合并 4 路召回结果
  • 同一张表被多路召回时，取各路权重之和作为该表的综合得分
  • 去除黑名单表：系统表、临时表、已下线表不进入推荐

Step 3：排序打分

每张候选表按以下公式计算最终得分：

综合得分 = 召回权重得分 × 0.5 + 热度分 × 0.2 + 新鲜度分 × 0.15 + 质量分 × 0.15
得分维度 分值范围 计算规则
召回权重得分 0 ~ 1 各路召回权重系数之和（被多路召回则叠加）
热度分 0 ~ 1 近 30 天查询次数归一化（min-max 归一到 [0,1]）
新鲜度分 0 ~ 1 表最后更新时间：7 天内=1.0，30 天内=0.6，超过 30 天=0.2
质量分 0 ~ 1 表元数据完整度：有描述 +0.3、有负责人 +0.3、有标签 +0.2、通过 DQ 检测 +0.2

Step 4：截断与输出

  • 按综合得分降序排列，取 Top 3~5 张表作为最终推荐结果
  • 若用户负责的表 ≤ 2 张（新用户），跳过血缘召回，角色画像召回权重提升至 0.4
  • 若总候选 < 3 张，以全局热度 Top 表补足至 3 张（保底策略）

推荐结果展示规则

场景 展示策略
正常推荐（≥3 张表） 展示 3~5 张表卡片 + "查看更多推荐表" 按钮
推荐不足（<3 张表） 展示所有候选 + 全局热度补足 + "查看更多推荐表" 按钮
完全无推荐数据 不展示表卡片，问候语改为引导文案："你可以试试以下方式找到需要的数据表 →"，突出意图卡片
"查看更多"跳转 跳转到 /search?q={department关键词}&source=copilot_recommend，搜索页自动执行推荐扩展查询

推荐刷新策略

触发时机 行为
新建对话 重新调用推荐接口，获取最新推荐列表
切换回已有会话 展示该会话创建时的推荐快照（不重新请求）
用户登录后首次打开 重新请求，确保推荐结果反映最新行为数据
推荐缓存 客户端缓存 5 分钟，避免短时间内重复请求

3.3 意图系统（四大智能找表意图）

意图系统是 Copilot "人找表"能力的核心，通过 4 种预设意图引导用户高效描述需求。

3.3.1 四大意图定义

序号 意图名称 意图 ID 图标 输入框占位提示
1 按业务用途找表 smart_find_business AppstoreOutlined "描述你的业务场景或分析目的..."
2 按关键词找表 smart_find_keyword SearchOutlined "输入表名、库名、字段名等关键词..."
3 指定项目范围找表 smart_find_project DeploymentUnitOutlined "描述项目名称和你要找的数据..."
4 指定专题范围找表 smart_find_topic TagsOutlined "指定数据专题，描述你的需求..."

3.3.2 意图交互流程

  1. 触发：用户点击欢迎区的意图卡片（完整模式）或意图药丸按钮（紧凑模式）
  2. 激活态：
    • 输入框左侧出现紫色意图标签（pill），显示意图名称，带关闭 x 按钮
    • 输入框 placeholder 切换为该意图的专属提示文案
    • 输入框自动聚焦，等待用户输入
  3. 发送：用户输入文本后按 Enter 或点击发送按钮
    • 消息携带 intentId 字段发送至后端
    • 发送后自动清除意图激活态（一次性使用）
  4. 取消：点击意图标签上的 x 按钮，清除意图激活态，恢复默认 placeholder

3.3.3 意图识别产品逻辑

用户发送消息时，后端需要完成"意图识别 → 参数提取 → 检索执行 → 结果组装"四个阶段。以下逐一定义每种意图的完整处理逻辑。

意图一：按业务用途找表 (smart_find_business)

阶段 逻辑说明
适用场景 用户不知道具体表名，只能描述业务需求。
例："我想分析渠道转化漏斗"、"查看上个月的退款数据"、"做用户留存分析需要哪些表"
意图识别 主动触发：用户点击意图卡片后输入，消息携带 intentId=smart_find_business
被动识别：用户直接输入自然语言（无 intentId），由 LLM 判断输入中包含业务场景描述 → 自动分类为此意图
识别信号词："分析""看看""查看""了解""哪些表能""什么数据可以"
参数提取 LLM 从自然语言中提取：
1. 业务域（business_domain）：交易/用户/营销/风控/财务/供应链/内容 等
2. 分析目的（analysis_purpose）：转化分析/留存分析/漏斗/趋势/对比/归因 等
3. 时间范围（time_range）：如 "上个月""最近7天"（可选）
4. 指标关键词（metric_keywords）：如 "GMV""DAU""转化率"（可选）
检索执行 1. 根据 business_domain 锁定候选表范围（按表的 domain/tag 标签过滤）
2. 用 analysis_purpose + metric_keywords 对候选表做语义相似度匹配（表描述 + 字段描述 + 业务术语表）
3. 按语义相似度得分降序排列
4. 取 Top 3~8 张表
结果组装 回复文本模板：
"根据你描述的「{analysis_purpose}」场景，我在 {business_domain} 域为你找到以下 {N} 张相关数据表："

附带表卡片列表（ResultCards），每张卡片额外展示匹配原因标签：
- "业务域匹配" / "描述相关" / "字段包含{keyword}" / "同部门高频使用"
兜底策略 若语义匹配结果 < 2 张表：
1. 降级为关键词搜索（提取输入中的名词作为关键词）
2. 若仍无结果，回复："暂未找到完全匹配的数据表，你可以换个描述方式试试，或者试试 按关键词找表。"

意图二：按关键词找表 (smart_find_keyword)

阶段 逻辑说明
适用场景 用户知道部分表名、库名、字段名或技术关键词。
例："order_detail"、"dwd 层的支付表"、"包含 user_id 字段的表"
意图识别 主动触发：intentId=smart_find_keyword
被动识别信号：输入中包含下划线命名（xxx_xxx）、分层前缀（dwd/dwa/dm/ads/ods）、"字段""列名""库""schema"等技术词汇
参数提取 1. 关键词列表（keywords）：按空格/逗号拆分，保留原始形态
2. 匹配范围（match_scope）：自动判断用户意图——
  - 包含下划线 → 优先匹配表名/字段名
  - 包含中文 → 优先匹配表描述/中文名/业务术语
  - 包含分层前缀 → 增加分层过滤条件
3. 分层过滤（layer_filter）：如 "dwd" → 仅搜索 DWD 层（可选）
检索执行 多维度并行检索，按权重合并：
1. 表名精确匹配（权重 0.4）：fullyQualifiedName / name LIKE '%keyword%'
2. 字段名匹配（权重 0.25）：字段 name LIKE '%keyword%'，返回所属表
3. 描述/中文名匹配（权重 0.2）：displayName / description 全文检索
4. 标签匹配（权重 0.15）：tag / glossary 术语表匹配

结果按综合相关度分排序，取 Top 5~10 张表
结果组装 回复文本模板：
"关键词 「{keywords}」 的搜索结果，共找到 {N} 张相关表{layerFilter ? '（限定 ' + layer + ' 层）' : ''}："

表卡片列表，每张卡片高亮匹配命中位置：
- 表名命中 → 表名中关键词加粗
- 字段命中 → 卡片下方额外显示 "匹配字段：{field_name}"
- 描述命中 → 描述文本中关键词高亮
兜底策略 若精确匹配无结果：
1. 尝试模糊匹配（编辑距离 ≤ 2 的拼写纠错）
2. 若仍无结果，回复："未找到包含「{keyword}」的数据表。建议检查拼写，或试试 按业务用途找表 用自然语言描述需求。"

意图三：指定项目范围找表 (smart_find_project)

阶段 逻辑说明
适用场景 用户只想在特定项目空间内检索表。
例："风控项目里有哪些用户画像表"、"交易中台项目下的 DWD 层表"
意图识别 主动触发：intentId=smart_find_project
被动识别信号："项目""project""空间""workspace" + 具体项目名称
参数提取 1. 项目名称（project_name）：LLM 从输入中提取，与项目列表做模糊匹配确认
2. 附加查询（sub_query）：项目范围确定后，输入中的剩余部分作为子查询
3. 若项目名模糊：返回候选项目列表让用户选择（"你是指以下哪个项目？"）
检索执行 1. 根据 project_name 获取该项目空间下所有已注册的表（project → databases → tables）
2. 若有 sub_query → 在项目范围内执行意图一（业务用途）或意图二（关键词）的检索逻辑（自动判断）
3. 若无 sub_query → 按热度排序返回该项目下 Top 10 的高频表
结果组装 回复文本模板：
"在项目 「{project_name}」 范围内{sub_query ? '，与「' + sub_query + '」相关的' : '热门'}数据表如下："

表卡片列表，每张卡片额外显示项目标签（蓝色 badge）
消歧交互 当项目名称匹配到多个项目时：
回复："找到多个相关项目，请确认你要查找的是哪一个：" + 项目名称列表（可点击选择）
用户点击后系统自动以该项目 + 原始 sub_query 重新检索
兜底策略 项目不存在：回复 "未找到名为「{project_name}」的项目，当前可用的项目有：{projectList}。"
项目下无表：回复 "项目「{project_name}」下暂无注册的数据表。"

意图四：指定专题范围找表 (smart_find_topic)

阶段 逻辑说明
适用场景 用户想在已有的数据专题内检索或浏览表。
例："交易分析专题里有哪些表"、"用户画像专题里和留存相关的表"
意图识别 主动触发：intentId=smart_find_topic
被动识别信号："专题""主题""topic""域" + 具体专题名称或业务域关键词
参数提取 1. 专题名称（topic_name）：LLM 提取，与专题列表做模糊匹配
2. 附加查询（sub_query）：专题确定后的附加过滤条件
3. 若专题名模糊：返回候选专题列表（含 tableCount 和 description）让用户选择
检索执行 1. 根据 topic_name 获取该专题下的所有关联表（topic.tables）
2. 若有 sub_query → 在专题表列表中做语义匹配或关键词匹配
3. 若无 sub_query → 按热度排序返回全部表（≤20 张直接返回，>20 张取 Top 20）
4. 权限过滤：若专题为私有（visibility='private'）且用户非 owner/admin → 返回权限不足提示
结果组装 回复文本模板：
"专题 「{topic_name}」（共 {tableCount} 张表）{sub_query ? '中与「' + sub_query + '」相关的' : '包含以下'}数据表："

表卡片列表 + 底部显示 "查看专题详情 >" 链接（跳转 /topics/{topicId}）
消歧交互 匹配到多个专题时：
回复："找到多个相关专题：" + 专题卡片列表（名称 + 描述 + 表数量），用户点击选择后重新检索
兜底策略 专题不存在：回复 "未找到名为「{topic_name}」的数据专题。当前热门专题有：{topicList}。"
权限不足：回复 "专题「{topic_name}」为私有专题，你暂无查看权限，请联系专题管理员 {owner}。"

意图识别优先级与冲突处理

当用户直接输入自然语言（无 intentId）时，LLM 需自动判断意图类型。以下是判断优先级和冲突处理规则：

优先级 判断条件 分类为 示例
1 明确提到"专题"/"主题" smart_find_topic "交易分析专题里有什么表"
2 明确提到"项目"/"空间" smart_find_project "风控项目下的用户表"
3 包含技术命名/下划线 smart_find_keyword "order_detail"、"dwd层支付"
4 纯自然语言业务描述 smart_find_business "我想看渠道转化数据"
5 无法判断意图 smart_find_business（兜底） 同时走关键词+语义双路检索
多轮对话上下文继承：在同一会话中，如果用户上一轮指定了项目/专题范围，当前轮未明确新范围时，默认继承上一轮的范围约束。用户可通过"换个范围""全局搜索"等表述主动解除范围限制。

3.3.4 AI 回复结构总览

所有意图的 AI 回复均遵循统一结构：

回复组成 说明 是否必须
范围确认文本 确认检索的范围和条件（如项目名/专题名/业务域），让用户确认理解正确 必须
匹配说明 简要说明找到多少张表、按什么排序 必须
表卡片列表 ResultCards 组件渲染，每张卡片含匹配原因标签 有结果时必须
追问引导 尾部添加引导文案：如"需要进一步了解某张表的详情或血缘关系吗？" 建议
兜底文案 无结果时的友好提示 + 替代方案建议 无结果时必须

3.3.4 推荐表卡片（ResultCards）

每张表卡片展示以下信息：

元素 说明 交互
表全限定名 格式：database.table_name，等宽字体，主色 点击跳转到 /detail/{fqn}
表中文名 displayName，灰色小字 -
数据源标签 SourceTag 组件（Hive / StarRocks / MySQL / ClickHouse 等），彩色 badge -
描述 单行截断，12px 灰色 hover 显示完整描述 tooltip
设计要点：卡片整体可点击，hover 态有边框高亮 + 微阴影提升。点击后在新 Tab 或当前页面跳转到表详情页（/detail/{fullyQualifiedName}）。

3.4 对话交互

3.4.1 消息类型与气泡样式

消息角色 对齐方式 气泡样式 头像 Hover 操作
用户消息 右对齐 紫色背景 rgba(108,76,155)，白色文字，圆角 14px 灰色圆形（#f0f0f0），用户首字母 编辑、复制
AI 消息 左对齐 白色背景，#e8ecf1 边框，圆角 14px PilotLogo 28px 紫色渐变 复制、点赞、点踩
Greeting 消息 左对齐 同 AI 消息，额外包含推荐表卡片 + "查看更多"按钮 同 AI 消息 同 AI 消息

3.4.2 AI 回复内容渲染

  • 文本：支持基础 Markdown 渲染（加粗 **text**），不支持完整 Markdown
  • 表卡片：AI 消息的 tables 字段不为空时，在文本下方渲染 ResultCards 组件
  • 打字机效果：AI 回复时先显示"正在思考"动画（3 个跳动圆点），600ms 后开始逐字输出

3.4.3 输入框交互

交互 行为
发送消息 Enter 发送（非空时）；Shift+Enter 换行
意图激活态 输入框左侧显示紫色 pill 标签（意图名称 + x 关闭），输入区宽度自适应收缩
空消息保护 输入内容为空（或仅空白字符）时，发送按钮禁用，Enter 无响应
AI 回复中 输入框可继续输入，但发送按钮变为 loading 态，不可重复发送
默认 Placeholder "输入你的问题，按 Enter 发送..."

3.4.4 用户消息编辑（覆盖重写 / 仅回退）

用户可对已发送的消息进行编辑，支持两种操作模式：

操作 行为 适用场景
覆盖重写 替换当前消息内容 → 删除该消息之后的所有消息 → 以新内容重新发送，触发 AI 回复 用户想修改提问并重新获取回答
仅回退 删除当前消息及其之后的所有消息，不触发新的 AI 回复 用户想撤回提问，回到之前的对话状态
取消 关闭编辑器，不做任何修改 误触编辑
编辑器 UI：点击用户消息的"编辑"按钮后，消息气泡原地替换为 3 行 textarea（预填原文并全选），下方显示"覆盖重写"、"仅回退"、"取消"三个操作按钮。按 ESC 等同于取消。

3.5 反馈机制（点赞 / 点踩）

3.5.1 反馈操作入口

  • 每条 AI 消息 hover 时，在气泡右下角显示操作栏：复制 | 点赞 | 点踩
  • 点赞和点踩互斥：点击其中一个会取消另一个
  • 再次点击同一个按钮可取消反馈（toggle 行为）

3.5.2 点赞

  • 点击后图标变为实心 + 主色高亮
  • 仅记录 feedback='like'，不展开任何表单

3.5.3 点踩 + 内联反馈卡片

点击"踩"后，在该消息气泡正下方平滑展开（slideDown 动画）内联反馈卡片，不使用全局弹窗，保持对话沉浸感。

表单元素 说明 必填
问题分类标签 6 个可多选的 Tag 按钮：
1. 找不到想要的表
2. 表/字段信息不准
3. 血缘关系错误
4. 意图理解偏差
5. 回答卡顿/不完整
6. 其他 		必填（至少选 1 个）
补充说明 Textarea 文本域，placeholder "请描述具体问题..." 选填，最多 150 字（实时字数统计）
操作按钮 "取消"（关闭卡片）+ "提交"（禁用态直到至少选 1 个分类） -
提交后行为：
  • 反馈卡片收起，踩图标保持高亮（表示已提交反馈）
  • 反馈数据（dislikeReasonIds + dislikeDetail）存储在该条消息对象上
  • 后续对接后端时，提交至 POST /v1/copilot/feedback 接口

3.6.1 会话生命周期

状态 标识 说明
