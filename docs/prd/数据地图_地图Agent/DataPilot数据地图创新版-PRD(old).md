# DataPilot 数据地图创新版 (MVP) - 产品需求文档 (PRD)

## 1. 产品概述 (Overview)

### 1.1 核心目标
打造一个以**“对话式交互”**为核心的数据地图平台，将传统数据地图的“搜索-查看-申请”流程重塑为智能对话体验。降低业务人员和数据开发人员的数据探索门槛，提升数据资产的发现、理解和复用效率。

### 1.2 问题定义
传统数据地图存在以下痛点：
- **门槛高**：需要掌握复杂的查询语法和系统操作流程。
- **效率低**：找表、看表、理解表、申请权限的流程割裂，需要跨多个页面跳转。
- **沉淀难**：分析过程和经验难以沉淀为团队共享的知识。
- **上下文割裂**：数据资产的元信息无法无缝地流转到其他开发或分析工具中。

### 1.3 产品定位与架构理念
**前台：对话即服务（面向业务与轻量探查）**
完全摒弃传统的多页面跳转，通过自然语言交互完成核心数据探索场景。前台极其克制，只提供对话、卡片和上下文分享，专注于“消费”元数据，提供极致的交互体验。**明确指出 DataPilot 是作为数据资产的“智能消费前台”，而元数据的维护、打标、质量规则配置等管理动作，将由独立的“数据地图管理后台（或 OpenMetadata 控制台）”承担。DataPilot 仅读取并展示这些高质量的元数据。**

**后台：依托 OpenMetadata（面向数据开发）**
后台不需要给业务用户查看，主要面向数仓和数据管理员。**底层元数据管理完全依托 OpenMetadata 开源产品承接**。复杂的元数据同步、打标等管理动作，深度融入数据开发和资产生产流程中。所需要的底层元信息结构不变（甚至更丰富），只是前台的消费和交互方式被彻底重构。

**核心理念：无孔不入的上下文提供 (Context as a Service) 与可插拔能力**
数据地图 Agent 不仅仅是一个独立的系统首页，它本质上是一个**可插拔的元数据消费引擎**。
1. **独立入口形态**：作为平台的主入口，提供沉浸式的全屏对话体验。
2. **嵌入式形态（可插拔 Copilot）**：将核心功能封装为**可独立引入的微前端组件（如 Copilot 侧边栏、独立卡片组件）**。允许无缝嵌入到 SQL 开发台、BI 报表等其他数据工作流中。支持**上下文注入**（如读取当前 SQL 代码）和**双向联动**（如一键将字段插入编辑器），实现“边工作，边探查”。
3. **跨系统形态**：通过 MCP (Model Context Protocol) 等标准协议，将这些元数据无缝提供给其他系统（如 Cursor/VSCode 等 AI 编码助手），实现元数据的全局复用。
**边界定位**：DataPilot 专注于提供标准化的对话 UI 组件、卡片组件、API 接口及意图识别逻辑，不负责开发宿主系统（如 SQL 编辑器），只作为“插件/副驾驶”被集成。

## 2. 用户故事与使用场景 (User Stories & Scenarios)

### 2.1 核心用户
- **数据消费者（业务/运营/分析师）**：需要快速找到业务相关的数据表，理解字段含义，并获取数据权限。
- **数据开发者**：需要快速探查表结构、质量情况，以及追踪数据血缘。

### 2.2 核心交互模式（双轨输出）
为了彻底替代传统数据地图的搜索与查看流程，对话框将提供两种深度的信息输出模式：

#### 模式一：快速信息模式（Quick Info Card）
- **触发场景**：用户进行明确的找表、查字段、看血缘等基础查询。
- **输出形态**：结构化的**静态卡片**（如表基础信息卡片、血缘图卡片）。
- **核心能力**：
  - 展示表的核心元数据（表名、负责人、描述、核心字段）。
  - 支持“查看全部”展开更多细节。
  - 支持类似千问的**“分享静态页面”**功能，用户可将该卡片/对话快照生成链接分享给同事。

#### 模式二：深度探查模式（Deep Insight Report）
- **触发场景**：用户需要深入了解某张表，或提出模糊的业务分析需求（如“帮我分析一下这张表的现状”）。
- **输出形态**：由 Agent 推理生成的**深度探查报告（长文本 + 多图表组合）**。
- **核心能力（Agent 秘书视角）**：
  - **上下文感知**：结合用户最近的行为（如最近常查的业务域）和身份（开发 vs 业务），调整报告侧重点。
  - **主动推理**：不仅提供静态数据，还主动提示风险与洞察。例如：“该表最近 3 天产出延迟，下游有 5 个重要报表受影响”、“该表空值率较高，建议关注”。
  - **全景信息**：整合数据质量、血缘影响面、高频查询 SQL 示例、业务口径等，生成一份“用户应该知道”的综合报告。

## 3. 页面结构与信息架构 (Information Architecture)

MVP 版本聚焦于以下核心界面层：

### 3.1 对话层 (Chat UI)
- **消息展示区**：用户消息气泡、AI 响应消息、卡片组件（表卡片、血缘图卡片）、流式打字指示器。
- **输入区**：意图选择器（可选）、文本输入框、`@` 提及功能、发送按钮。
- **侧边栏**：新建对话、历史会话列表。

### 3.2 工作台 (Workspace)
- **推荐数据表**：基于访问频率、最近使用、协同过滤推荐。
- **最近对话**：对话列表，支持点击继续对话。
- **收藏的数据表**：快速访问常用表。

### 3.3 数据专题 (Topics)
- **对话快照**：完整对话记录、涉及的表、生成的 SQL、分析结论、分享功能。
- **智能表集合**：表列表、AI 生成主题、关联说明、快速启动。
- **分析模板**：模板列表、应用模板、自定义模板。

## 4. MVP 范围界定 (MVP Scope)

### 4.1 核心功能层 (In Scope)
- **对话内表详情展示 (不跳出对话流)**：
  - **基础信息卡片**：在对话流中直接返回表的基础信息（表名、负责人、描述、核心字段预览）。
  - **查看全部 (抽屉交互)**：点击卡片上的“查看全部”，在对话流右侧弹出抽屉（Drawer）展示完整的表详情（含完整字段、血缘、生产信息等），**不离开当前对话页面**。
- **静态分享功能 (分享颗粒度)**：
  - **单卡片分享**：支持将单个表详情卡片生成静态只读 Web 链接分享给同事。
  - **会话上下文分享**：支持将整段对话上下文（包含提问和多个卡片）生成静态链接分享。
- **权限管理 (弱管控模式)**：
  - 默认允许所有用户查看数据表的元数据详情。
  - 默认允许所有用户查看**前 20 条样例数据**（数据预览）。
  - 仅在用户需要执行深度操作（如发起真实查询、导出全量数据）时，才进行严格权限校验并提供“一键申请”入口。**（注：这里的申请是推送到后台管理系统或 OA 系统进行审批，DataPilot 前台仅负责发起动作和展示工单状态，不包含复杂的审批流配置界面。）**
- **资产搜索**：按业务用途、关键词、项目范围、专题范围找表，展示搜索结果卡片。
- **血缘关系**：上游来源、下游引用、血缘图可视化、血缘洞察。

### 4.2 智能服务层 (In Scope)
- **智能推荐**：推荐引擎（历史、最近、协同）、推荐展示（工作台卡片、对话追问建议）。
- **对话历史管理**：会话列表（按时间倒序、搜索）、会话操作（继续、重命名、删除、导出）。

### 4.3 产品特点 (Experience Requirements)
- **自然语言交互**：降低使用门槛，理解业务场景。
- **实时流式响应**：降低等待焦虑，透明化处理过程。
- **智能记忆**：个性化体验，减少重复输入。

### 4.4 非目标 (Out of Scope for MVP)
- **Agent 深度探查报告**：基于用户行为和上下文主动生成的深度推理报告（如血缘风险预警、质量突变分析、协同上下文提示等），作为未来演进方向，不在本次 MVP 范围内。
- 复杂的传统数据大盘（Dashboard）和多维筛选列表页（保留为后续或 V2 版本能力）。
- 复杂的数据开发 IDE 能力。
- 复杂的后台管理配置页面。

## 5. 验收标准与成功指标 (Acceptance Criteria)

- **核心链路通畅**：用户能够通过自然语言成功搜索到目标表，并在对话流中查看表详情和血缘图。
- **体验达标**：流式输出平滑，卡片渲染无明显卡顿；`@` 提及功能能够准确唤出资产列表。
- **双轨策略**：
  - [ ] **原型轨**：是否需要先输出 HTML 原型？（待确认：是/否跳过）
  - [ ] **工程轨**：基于 Vue3 + Antdv + Tailwind 构建业务组件库，组件拆解符合 L1/L2/L3 规范。
