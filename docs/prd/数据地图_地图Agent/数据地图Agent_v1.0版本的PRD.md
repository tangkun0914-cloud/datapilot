## 一、产品概述

### 1.1 产品定位

DataPilot 是一款**对话式数据分析平台**，通过自然语言交互的方式，帮助用户快速查询数据表信息、理解数据血缘关系、申请数据权限，降低数据发现和使用门槛。

**核心价值：**
- 🔍 **快速找表**：通过自然语言描述业务场景，智能推荐相关数据表
- 💬 **对话式交互**：无需学习复杂的查询语法，像聊天一样获取数据信息
- 🤖 **智能推荐**：基于用户行为和协同过滤，推荐常用数据表
- 🔗 **血缘可视化**：内联血缘图，直观展示数据的上下游关系
- 📤 **知识分享**：支持对话分享，沉淀和传播数据分析经验
- 🔌 **可插拔能力**：作为无处不在的 Copilot，不仅提供独立首页，还可作为微前端组件嵌入 SQL 查询台、BI 报表等业务系统，支持上下文注入与双向联动，实现“边工作，边探查”。
- 🔌 **MCP 集成**：为 Cursor 等外部工具提供数据上下文能力

**产品边界定位：**
明确指出 DataPilot 是作为数据资产的**“智能消费前台”**，而元数据的维护、打标、质量规则配置等管理动作，将由独立的“数据地图管理后台（或 OpenMetadata 控制台）”承担。DataPilot 仅读取并展示这些高质量的元数据。DataPilot 专注于提供标准化的对话 UI 组件、卡片组件、API 接口及意图识别逻辑，不负责开发宿主系统（如 SQL 编辑器），只作为“插件/副驾驶”被集成。

### 1.2 目标用户

| 用户角色 | 典型场景 | 核心诉求 |
|---------|---------|---------|
| **数据分析师** | 业务分析前需要找到合适的数据表 | 快速找到业务相关的表，了解字段含义和数据质量 |
| **算法工程师** | 特征工程需要了解数据血缘 | 查看表的上游来源，评估数据可靠性 |
| **产品经理** | 业务需求调研需要了解数据资产 | 无需技术背景，通过自然语言快速了解数据概况 |
| **开发工程师** | 在 IDE 中编码时需要数据表上下文 | 通过 MCP 协议在 Cursor 中直接获取表信息 |

| 特点 | 说明 | 价值 |
|-----|------|------|
| **自然语言交互** | 支持业务场景描述，无需学习查询语法 | 降低使用门槛，提升效率 |

## 二、MVP 范围界定

### 2.1 MVP 核心场景

MVP v1.0 聚焦**快速信息模式**，通过对话式静态卡片展示，帮助用户快速获取数据表信息。

**核心场景：**
1. ✅ **查询数据表详情**：通过自然语言查询表的基本信息、字段、数据预览、质量评估、建表脚本
2. ✅ **智能搜索数据表**：支持按业务用途、关键词、项目范围、专题范围找表
3. ✅ **查看血缘关系**：可视化展示表的上下游关系
4. ✅ **权限管理**：自动检测权限，引导用户一键发起申请（注：这里的申请是推送到后台管理系统或 OA 系统进行审批，DataPilot 前台仅负责发起动作和展示工单状态，不包含复杂的审批流配置界面。）
5. ✅ **对话分享**：支持单条对话和完整会话分享
6. ✅ **MCP Server**：为 Cursor 等外部工具提供数据上下文

**不在 MVP 范围内（延后到 V2）：**
- ❌ 智能探查模式（Agent 深度推理）
- ❌ 数据专题（对话快照、智能表集合、分析模板）
- ❌ 上下文管理集成
- ❌ SQL 生成与执行
- ❌ 数据分析图表生成

## 三、用户界面设计

### 3.1 整体布局

DataPilot 采用**对话优先架构**，主入口是对话界面，工作台作为侧边栏面板。

**布局结构：**
```
┌─────────────────────────────────────────────────┐
│  [Logo] DataPilot 数据地图          [用户头像]  │
├──────────┬──────────────────────────────────────┤
│          │                                      │
│  侧边栏   │         对话区域                      │
│          │                                      │
│ [新建对话] │  ┌─────────────────────────────┐   │
│          │  │  你好，我是数据地图 Agent    │   │
│ 历史对话   │  │                            │   │
│ - 对话1   │  │  [大输入框]                 │   │
│ - 对话2   │  │  placeholder: 请输入你的问题 │   │
│          │  │  支持: Shift+Enter 换行     │   │
│ 工作台    │  │        Enter 发送           │   │
│ - 推荐表  │  └─────────────────────────────┘   │
│ - 最近对话│                                      │
│ - 收藏表  │                                      │
│          │                                      │
└──────────┴──────────────────────────────────────┘
```

**设计说明：**
- **对话优先**：页面中央是对话区域，首次进入显示问候语 + 大输入框
- **侧边栏双 Tab**：历史对话 + 工作台，用户可快速切换
- **大输入框**：首页输入框高度 120px，支持多行输入，提升交互体验
- **问候语**：显示 "你好，我是数据地图 Agent" + DataPilot Logo（8 角星渐变）

### 3.2 对话界面

#### 3.2.1 意图选择器（可选）

用户可以在输入框上方选择意图，帮助系统更准确地理解需求。

**4 种意图：**
1. 🔍 **搜索数据表**：我想找表
2. 📋 **查看表详情**：我想看某个表的详细信息
3. 🔗 **查看血缘关系**：我想了解表的上下游
4. 🔐 **申请数据权限**：我想申请某个表的权限

**交互方式：**
- 默认不选择，系统自动识别意图
- 用户可手动选择意图，提升识别准确率
- 选择后，输入框 placeholder 会变化（如："请输入表名或关键词"）

## 四、核心交互流程

### 4.1 快速信息模式（MVP 核心）

**场景：** 用户想快速了解某个表的基本信息。

**流程：**
```
用户输入："user_behavior_log 这个表是干什么的？"
   ↓
意图识别：查看表详情
   ↓
调用 OpenMetadata API：获取表元数据
   ↓
流式输出（SSE）：
  1. "正在查询表信息..."
  2. "找到了表 user_behavior_log"
  3. [表卡片展示]
     - 基本信息：负责人、更新频率
     - 字段概览：15 个字段（前 3 个）
     - 质量评分：98 分
  4. [查看全部] [分享] [收藏]
   ↓
追问建议：
  - "查看完整字段列表"
  - "查看血缘关系"
  - "查看数据预览"
```

**关键点：**
- **卡片展示**：核心信息折叠展示，点击"查看全部"跳转详情页
- **流式输出**：边查询边输出，降低等待焦虑
- **追问建议**：AI 生成下一步操作建议

---

### 4.2 搜索数据表流程

**场景：** 用户不知道表名，通过业务场景描述找表。

**流程：**
```
用户输入："我想分析用户的购买行为，需要哪些表？"
   ↓
意图识别：按业务用途搜索
   ↓
```

#### 5.3.3 推荐引擎

**职责：**
- 访问频率统计：基于用户历史访问记录
- 最近使用排序：基于时间戳排序
- 协同过滤：基于同事的访问行为推荐

**推荐算法：**
```
推荐分数 = 0.4 × 访问频率 + 0.3 × 最近使用 + 0.3 × 协同过滤

访问频率 = 访问次数 / 总访问次数
最近使用 = 1 / (当前时间 - 最后访问时间)
协同过滤 = 同事访问次数 / 同事总访问次数
```

**API 设计：**
```javascript
GET /api/recommend/tables?user_id=123&limit=10

Response:
{
  "recommendations": [
    {
      "table_name": "user_behavior_log",
      "score": 0.92,
      "tags": ["高频使用", "协同推荐"]
    }
  ]
}
```

### 5.5 缓存策略

**缓存层次：**

| 层次 | 缓存内容 | TTL | 策略 |
|-----|---------|-----|------|
| **L1 - 内存缓存** | 热点表元数据 | 5分钟 | LRU |
| **L2 - Redis 缓存** | 表详情、血缘关系 | 1小时 | TTL + 手动刷新 |
| **L3 - OpenMetadata** | 完整元数据 | 永久 | 定时采集 |

**缓存更新策略：**
- **主动刷新**：用户点击"刷新"按钮，清除缓存
- **被动刷新**：OpenMetadata 数据更新后，通过 Webhook 通知 DataPilot
- **定时刷新**：每天凌晨 2 点，全量刷新缓存

## 七、风险管理

### 7.1 风险识别

| 风险 | 影响程度 | 发生概率 | 应对方案 |
|-----|---------|---------|---------|
| **OpenMetadata 部署延迟** | 🔴 高 | 中 | 先用 Mock 数据开发，并行推进 OpenMetadata 部署 |
| **LLM API 调用限流** | 🟡 中 | 高 | 增加缓存，控制调用频率，准备备用 API Key |
| **推荐算法效果不佳** | 🟡 中 | 中 | 先用简单规则（访问频率 + 最近使用），V2 优化算法 |
| **MCP 协议理解不足** | 🟡 中 | 中 | 提前研究文档，找示例代码，必要时咨询社区 |
| **性能不达标** | 🟡 中 | 低 | 增加缓存，优化 SQL，必要时扩容服务器 |
| **数据质量不佳** | 🟠 中 | 中 | OpenMetadata 配置质量规则，定期检查 |
| **权限系统集成复杂** | 🟠 中 | 低 | 简化 MVP 权限逻辑，V2 完善 |
