## 一、调研背景

团队当前基于 **OpenMetadata** 搭建元数据管理底座，并在此之上构建独立的 **上下文管理** 产品，服务 DataAgent / Text2SQL 等 AI 消费方。本文梳理 GitHub 上主流及新兴的开源元数据管理项目，重点关注：

- 元数据管理核心能力（目录、血缘、质量、Glossary）
- **AI / LLM 上下文交付能力**（MCP、语义层、API-first）
- 对我们产品设计的借鉴意义

## 三、重点项目详细分析

### 3.1 DataHub（LinkedIn）— 最成熟的 AI 集成

- **仓库：** [datahub-project/datahub](https://github.com/datahub-project/datahub)
- **Stars：** ~11,800
- **技术栈：** Java 后端 / React+TS 前端 / Elasticsearch / MySQL / Kafka

**核心能力：**

| 能力 | 说明 |
|------|------|
| 通用搜索 | 跨数据源的统一搜索与发现 |
| 列级血缘 | 自动解析和可视化 |
| 实时元数据更新 | 基于 Kafka 的流式更新 |
| 80+ 连接器 | 生产级连接器覆盖主流数仓 |
| **MCP Server** | **内置 MCP 协议支持，AI Agent 可直接查询元数据** |
| 治理看板 | 策略、标签、合规管理 |

**对我们的启发：**
- DataHub 的 MCP Server 实现是最直接的参考——它展示了如何把 schema、血缘、Glossary 通过 MCP 暴露给 Claude/Cursor 等 Agent
- GraphQL + REST + SDK 三种 API 并行的策略值得借鉴

---

### 3.2 OpenMetadata — 我们的底座

- **仓库：** [open-metadata/OpenMetadata](https://github.com/open-metadata/OpenMetadata)
- **Stars：** ~10,100
- **技术栈：** TypeScript (43.6%) / Java (34.2%) / Python (20.2%) / Elasticsearch / MySQL

**核心能力：**

| 能力 | 说明 |
|------|------|
| No-code 数据质量 | 内置 Profiler + Quality 框架，无需写代码 |
| 数据可观测性 | 新鲜度、数据量、延迟监控 |
| Glossary & Tags | 业务术语表 + 分类标签体系 |
| Data Domains | 数据域 & 数据产品抽象 |
| JSON Schema 驱动 | 所有元数据模型基于 JSON Schema，API 自动生成 |
| 84+ 连接器 | 覆盖 Hive、Spark、Doris、ClickHouse 等 |

**我们的定位：** OpenMetadata 提供元数据采集、存储、治理的基础设施，我们在此之上构建独立的 **上下文管理** 产品，做 AI 上下文的加工、组装、交付。

---

### 3.3 WrenAI — Text2SQL 语义层的最佳参考

- **仓库：** [Canner/WrenAI](https://github.com/Canner/WrenAI)
- **Stars：** ~14,900
- **技术栈：** TypeScript/Next.js (64%) / Python/FastAPI / Rust 查询引擎 / Qdrant 向量检索

**核心能力：**

| 能力 | 说明 |
|------|------|
| MDL 语义建模层 | 将 schema、指标、JOIN、业务定义编码为结构化模型 |
| Text2SQL | 自然语言 → SQL，语义层兜底准确率 |
| GenBI | 自然语言 → 图表 |
| 12+ 数据源 | BigQuery、Snowflake、MySQL、PostgreSQL 等 |
| 多 LLM | OpenAI / Claude / Gemini / Ollama |

**MDL（Modeling Definition Language）核心思路：**

```
传统做法：直接把 DDL 塞给 LLM → 准确率低
WrenAI 做法：schema + 指标口径 + JOIN 路径 + 业务定义 → 编码为 MDL → LLM 基于 MDL 生成 SQL
```

**对我们的启发：**
- **MDL 模式与我们"上下文组装"模块的理念高度一致** —— 不是把原始元数据一股脑塞给 LLM，而是加工成结构化的语义层
- WrenAI 用 Qdrant 做语义检索，我们的"三路召回"（关键词 + 语义 + 图检索）是更进一步的方案
- WrenAI 的 MDL 是静态定义的，我们的上下文组装是动态的 —— 按意图、按 Token 预算实时组装

---

### 3.4 Marmot — 定位最接近我们的新兴项目

- **仓库：** [marmotdata/marmot](https://github.com/marmotdata/marmot)
- **Stars：** ~537
- **技术栈：** Go (53%) / Svelte+TS 前端 / PostgreSQL

**核心能力：**

| 能力 | 说明 |
|------|------|
| 单 binary 部署 | Go 编译，零运维负担 |
| 全文 + 结构化搜索 | 数据发现 |
| 交互式血缘图 | 可视化探索 |
| Glossary + 所有权 | 业务治理 |
| **原生 MCP Server** | **明确定位为 "AI 的上下文层"** |

**对我们的启发：**
- Marmot 虽然体量小，但产品定位与我们的上下文管理高度重合
- 它证明了"元数据 as AI context layer"这个方向有明确需求
- 我们的差异化在于：更完整的加工链路（标准化 → 向量化 → 组装 → Token 预算管理）

### 3.5 其他值得关注的项目

#### Amundsen（Lyft）
- **仓库：** [amundsen-io/amundsen](https://github.com/amundsen-io/amundsen) | ~4.8k Stars
- **亮点：** 基于使用频率做 PageRank 排序，"越常用的表排越前"
- **借鉴：** 使用频率信号可以作为我们上下文排序的因子之一（模块③ 排序因子中的"使用频率 10%"）

#### Unity Catalog（Databricks → LF AI）
- **仓库：** [unitycatalog/unitycatalog](https://github.com/unitycatalog/unitycatalog) | ~3.4k Stars
- **亮点：** 多格式（Delta/Iceberg/Hudi）、多引擎、含 AI 模型资产管理
- **借鉴：** "统一目录管理数据 + AI 模型"的思路，未来我们也可能需要管理 Prompt 模板、Agent 配置等 AI 资产

#### Apache Gravitino
- **仓库：** [apache/gravitino](https://github.com/apache/gravitino) | ~2.9k Stars
- **亮点：** 联邦式元数据湖，跨区域跨云统一 API
- **借鉴：** 如果团队未来有多集群、多区域的元数据管理需求，这是可参考的联邦架构

#### OpenLineage（规范）
- **仓库：** [OpenLineage/OpenLineage](https://github.com/OpenLineage/OpenLineage) | ~2.4k Stars
- **亮点：** 血缘互操作的行业标准，Spark/Airflow/dbt/Flink 均有集成
- **借鉴：** 建议我们的血缘接入层兼容 OpenLineage 规范，降低多管道工具的接入成本

#### Marquez（LF AI & Data）
- **仓库：** [MarquezProject/marquez](https://github.com/MarquezProject/marquez) | ~2.2k Stars
- **亮点：** OpenLineage 参考实现，专注血缘追踪，GraphQL + REST
- **借鉴：** 可作为专门的血缘后端参考

#### SQLLineage
- **仓库：** [reata/sqllineage](https://github.com/reata/sqllineage) | ~1.6k Stars
- **亮点：** Python SQL 血缘解析库，从 SQL 语句中提取表级 + 列级血缘
- **借鉴：** 可嵌入我们的产品，对 LLM 生成的 SQL 做自动血缘提取

## 四、市场空白与我们的定位

### 4.1 当前竞品格局

```
                    元数据管理能力
                    （目录/血缘/质量/Glossary）
                         │
                 强 ─────┤
                         │
            OpenMetadata ●─────● DataHub
                         │              ↑ 已有 MCP
            Amundsen ●   │
                         │
                 中 ─────┤
                         │
            Gravitino ●  │    ● Marmot (轻量+MCP)
                         │
                 弱 ─────┤
                         │    ● WrenAI (语义层强)
                         │
                         └──────┬──────┬──────┬─── AI/LLM 上下文交付能力
                               弱     中     强
```

### 4.2 空白区域

**目前没有任何项目同时做到：**

1. 完整的元数据管理（目录 + 血缘 + 质量 + Glossary）
2. 多来源上下文接入（元数据 + 文档 + 示例 SQL + 业务规则）
3. 智能上下文组装（意图理解 + 三路召回 + Token 预算管理）
4. 标准化上下文交付（MCP 协议 + REST API）

- **DataHub** 有 MCP 但上下文组装能力弱（直接暴露原始元数据，没有按意图动态组装）
- **WrenAI** 有语义层但只服务 Text2SQL，不是通用的上下文管理平台
- **Marmot** 定位对但体量小、能力不完整

### 4.3 我们的差异化

```
OpenMetadata（元数据底座）
    +
上下文管理（我们的独立产品）
    =
唯一一个同时具备：
    ✓ 完整元数据管理能力（依托 OpenMetadata）
    ✓ 多来源上下文接入（不只元数据）
    ✓ 智能组装（意图理解 + 三路召回 + Token 预算）
    ✓ MCP 标准化交付
    ✓ 质量 & 治理闭环
```
