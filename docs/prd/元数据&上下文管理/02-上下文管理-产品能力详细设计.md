## 模块 ① 上下文接入（Source Management）

### 1.1 模块定位

> 把各种来源的"原料"通过统一的连接器收进来，让上下文管理有东西可加工。

上下文接入是整个产品的入口。没有原料，后面的加工、组装、交付都无从谈起。

### 1.2 接入的来源类型

上下文管理的原料不只来自元数据，需要支持多种来源类型：

| 来源类型 | 内容示例 | 提供方 | 优先级 |
|---------|---------|--------|--------|
| **结构化元数据** | 表结构、字段类型、分区策略、血缘关系 | 元数据管理（OpenMetadata API） | **P0** |
| **业务语义** | Glossary 术语、字段业务含义、指标口径定义 | 元数据管理（Glossary）+ 业务团队 | **P0** |
| **示例 SQL** | 经过验证的"自然语言 → SQL"对 | 业务团队 / 历史查询日志 | **P0** |
| **业务文档** | 数据字典、需求文档、Wiki 知识库 | Confluence / 飞书文档 / 本地文档 | P1 |
| **业务规则** | 计算逻辑、数据权限规则、命名规范 | 业务团队 / 规则引擎 | P1 |
| **查询日志** | 用户历史查询、高频查询模式 | DataAgent 使用日志 | P2 |

> **为什么"示例 SQL"是 P0？** 业界实践表明，经过验证的 NL→SQL 对是对 Text2SQL 准确率提升最大的单一上下文类型。相比表结构描述，few-shot 示例能让 LLM 更直接地理解"这个业务场景应该怎么写 SQL"。

### 1.3 核心产品能力

#### 1.3.1 连接器管理

| 能力 | 说明 |
|------|------|
| **连接器注册** | 管理员可配置数据来源连接信息（API 地址、认证方式、同步策略） |
| **连接器模板** | 预置常用连接器模板：OpenMetadata、飞书文档、Confluence、本地文件 |
| **连接测试** | 配置后可测试连接是否正常 |
| **连接器状态** | 可查看每个连接器的运行状态、最近同步时间、错误日志 |

#### 1.3.2 同步策略

| 策略 | 适用场景 | 说明 |
|------|---------|------|
| **全量同步** | 首次接入、数据源结构大变更 | 拉取来源全部数据 |
| **增量同步** | 日常运行 | 只拉取变更部分，减少开销 |
| **变更检测** | 元数据来源 | 检测 Schema 变更（新增表/字段、字段类型变更、表删除） |
| **定时调度** | 所有来源 | 可配置同步频率：每小时 / 每天 / 自定义 Cron |
| **手动触发** | 紧急更新 | 管理员可手动触发某个来源的同步 |

#### 1.3.3 来源溯源

每条上下文数据都必须带有来源信息：

```json
{
  "source_type": "openmetadata",
  "source_id": "om_connector_01",
  "source_path": "glossary/trade/GMV",
  "synced_at": "2026-04-10T10:30:00Z",
  "version": "v3"
}
```

**为什么需要溯源？**
- 出了问题能追到"这条上下文是从哪来的"
- 来源数据更新时，能精确知道哪些上下文需要重新加工
- 多来源描述同一实体时，能按来源权威度做冲突处理

### 1.4 产品界面（概要）

| 页面 | 功能 |
|------|------|
| **来源列表** | 查看所有已接入的数据来源，状态一目了然 |
| **新增来源** | 选择来源类型 → 填写连接信息 → 测试 → 启用 |
| **来源详情** | 查看同步历史、数据量统计、错误日志 |
| **同步配置** | 设置同步策略、频率、字段映射 |

### 1.5 与元数据管理的对接

上下文接入从元数据管理获取数据的方式：

```
元数据管理（OpenMetadata）
    │
    │  REST API
    │  /api/v1/tables          → 表结构、字段
    │  /api/v1/lineage         → 血缘关系
    │  /api/v1/glossaryTerms   → 业务术语
    │  /api/v1/dataQuality     → 质量信息
    │
    ▼
上下文接入（OpenMetadata Connector）
    │
    │  标准化为统一格式
    ▼
上下文存储
```

**元数据管理是上下文管理最重要的数据源，但通过 API 解耦，不是直接访问底层数据库。**

## 模块 ② 上下文加工（Processing & Enrichment）

### 2.1 模块定位

> 把接入的原始数据加工成 LLM 能高效消费的格式。原料变半成品。

接入层收到的是"生数据"——格式各异、粒度不同、可能有冗余和冲突。加工层的职责是把这些数据统一、增强、索引，让下游组装时能快速精准地找到需要的上下文。

### 2.2 核心产品能力

#### 2.2.1 内容标准化

不同来源的数据格式不同，需要统一成标准的上下文单元（Context Unit）：

**上下文单元结构：**

```json
{
  "id": "ctx_001",
  "type": "table_schema",
  "entity": "dm_trade.dws_order_summary_nd",
  "title": "订单汇总表 - 表结构",
  "content": "表名: dws_order_summary_nd\n库名: dm_trade\n中文名: 订单汇总表\n...",
  "metadata": {
    "source": "openmetadata",
    "entity_type": "table",
    "fqn": "dm_trade.dws_order_summary_nd",
    "tags": ["trade", "core"],
    "freshness": "2026-04-10T08:00:00Z"
  },
  "embedding": [0.012, -0.034, ...],
  "token_count": 256
}
```

**上下文单元类型：**

| type | 含义 | 来源 | 示例 |
|------|------|------|------|
| `table_schema` | 表结构信息 | 元数据管理 | 表名、字段列表、字段类型、分区键 |
| `column_description` | 字段业务描述 | 元数据管理 Glossary | "order_amt: 订单金额，单位分，不含退款" |
| `table_relationship` | 表间关系 | 元数据管理血缘 | "dwd_order_detail → dws_order_summary（聚合）" |
| `business_term` | 业务术语 | Glossary | "GMV = 已支付订单金额，不含取消和退款" |
| `example_sql` | 示例查询 | 人工标注 / 查询日志 | "查近7天GMV → SELECT SUM(gmv) FROM ... WHERE ..." |
| `business_rule` | 业务规则 | 业务团队 | "风控拦截订单不计入 GMV" |
| `document_chunk` | 文档片段 | Wiki / 飞书 | 数据字典中某个章节 |

#### 2.2.2 文档切片 & 向量化

对于非结构化内容（文档、业务规则描述等），需要切片和向量化以支持语义检索：

| 步骤 | 说明 |
|------|------|
| **智能切片** | 按语义边界切分，而非固定长度。表结构按"表"为单位，文档按"段落/章节"为单位 |
| **Embedding 生成** | 每个上下文单元生成向量表示，存入向量索引 |
| **Token 计数** | 预计算每个单元的 Token 数，供组装时做预算管理 |
| **去重检测** | 同一内容从不同来源接入时，检测并标记重复 |

**切片策略按类型不同：**

| 上下文类型 | 切片策略 |
|-----------|---------|
| 表结构 | 一张表 = 一个单元（含所有字段） |
| 字段描述 | 如果字段多（>50），按"核心字段"和"扩展字段"拆分 |
| 血缘关系 | 一条血缘链路 = 一个单元 |
| 示例 SQL | 一个 NL→SQL 对 = 一个单元 |
| 文档 | 按章节/段落切分，保持语义完整性 |

#### 2.2.3 Schema 增强

对元数据来源的表结构信息做自动增强，提升上下文信息密度：

| 增强项 | 说明 | 价值 |
|--------|------|------|
| **样本值填充** | 自动采样每个字段的典型值（top 5） | LLM 能理解字段的实际数据长什么样 |
| **值分布统计** | NULL 比例、唯一值数量、最大/最小值 | LLM 能判断字段是否适合做 WHERE/GROUP BY |
| **关联路径标注** | 标注常用 JOIN 路径（基于血缘和查询日志） | Text2SQL 时 LLM 知道表该怎么 JOIN |
| **字段分类标签** | 主键、外键、维度、度量、时间分区 | LLM 能快速定位关键字段 |

**增强后的表结构示例：**

```
表名: dws_order_summary_nd
中文名: 订单汇总表
分区: dt (按天分区)
核心字段:
  - order_id (STRING) [主键] 订单编号  样本值: ORD20260401001, ORD20260401002
  - order_amt (BIGINT) [度量] 订单金额(分)  范围: 1 ~ 9999900  NULL率: 0%
  - pay_status (INT) [维度] 支付状态  枚举: {0:未支付, 1:已支付, 2:已退款}
  - dt (STRING) [分区键] 数据日期  格式: yyyyMMdd
常用 JOIN:
  - JOIN dm_trade.dwd_order_detail_di ON order_id（明细展开）
  - JOIN dm_trade.dim_pay_type ON pay_type_id（关联支付方式）
```

#### 2.2.4 去重与冲突处理

多来源可能描述同一实体，需要处理冲突：

| 场景 | 处理策略 |
|------|---------|
| **同一字段，多个描述** | 按来源权威度排序：Glossary > 文档 > 自动生成 |
| **过时描述** | 以最新同步时间为准，旧描述标记为"历史版本" |
| **矛盾信息** | 标记冲突，通知管理员人工仲裁 |

**来源权威度默认排序：**

```
元数据管理（Glossary）> 人工标注 > 业务文档 > 查询日志推断 > 自动生成
```

管理员可调整权威度排序。

#### 2.2.5 时效性管理

| 能力 | 说明 |
|------|------|
| **新鲜度追踪** | 每个上下文单元记录最后更新时间 |
| **过期策略** | 可配置过期阈值（如 7 天未更新标记为"可能过期"） |
| **级联更新** | 上游来源变更时，自动触发相关上下文单元的重新加工 |
| **变更通知** | Schema 变更（表/字段增删改）时，主动通知下游 |

### 2.3 加工流水线

一条上下文数据从接入到可用的完整流程：

```
原始数据（接入层）
    │
    ▼
[1] 格式解析 — 识别来源格式，提取内容
    │
    ▼
[2] 标准化 — 转为统一的上下文单元格式
    │
    ▼
[3] 增强 — Schema 增强（样本值、值分布、关联路径）
    │
    ▼
[4] 切片 — 按策略切分为合适粒度
    │
    ▼
[5] 向量化 — 生成 Embedding + 计算 Token 数
    │
    ▼
[6] 去重 & 冲突检测 — 标记重复，处理矛盾
    │
    ▼
[7] 写入存储 — 结构化存储 + 向量索引 + 全文索引
    │
    ▼
上下文单元（可被组装层消费）
```

### 2.4 产品界面（概要）

| 页面 | 功能 |
|------|------|
| **上下文总览** | 已加工的上下文单元统计：按类型分布、按来源分布、总量趋势 |
| **上下文浏览** | 按表/按实体查看关联的所有上下文单元，预览内容 |
| **加工配置** | 配置切片策略、增强规则、权威度排序 |
| **加工日志** | 查看加工流水线的执行记录、成功/失败/跳过统计 |
| **冲突处理** | 查看标记为"矛盾"的上下文，人工仲裁 |

## 模块 ③ 上下文组装（Context Assembly） — 核心差异化

### 3.1 模块定位

> 根据消费方的请求，从已加工的上下文单元中**动态选择、排序、裁剪、拼装**出最合适的上下文包。

这是整个上下文管理产品**最核心的能力**，也是与"简单 RAG"拉开差距的地方。

**核心理念：不是把所有上下文一股脑塞给 LLM，而是精准投喂。**

举例：用户问"近7天 GMV 是多少"，LLM 需要的上下文是：
- `dws_order_summary_nd` 的表结构（知道有哪些字段）
- `gmv` 字段的业务口径（知道 GMV 怎么算）
- 分区字段 `dt` 的格式（知道怎么筛时间）
- 一条类似的示例 SQL（知道该怎么写）

而**不需要**：其他 200 张表的结构、血缘全链路、数据质量报告。

### 3.2 组装的完整流程

```
消费方请求
    │
    ▼
[1] 意图理解 — 这个请求需要什么类型的上下文？
    │
    ▼
[2] 候选召回 — 从存储中召回可能相关的上下文单元
    │
    ▼
[3] 相关性排序 — 对候选集按相关性打分、排序
    │
    ▼
[4] Token 预算裁剪 — 在窗口限制内选出最高价值的组合
    │
    ▼
[5] 格式组装 — 按场景模板拼装成最终的上下文包
    │
    ▼
上下文包（交付给消费方）
```

以下逐步展开。

### 3.3 意图理解（Intent Understanding）

不同的请求需要不同的上下文"配方"。组装的第一步是理解消费方要什么。

#### 3.3.1 意图类型与上下文配方

| 意图 | 触发场景 | 需要的上下文类型 | 优先级排序 |
|------|---------|----------------|-----------|
| **Text2SQL** | 用户用自然语言查数据 | 表结构 > 示例SQL > 业务术语 > 字段描述 > JOIN路径 | 示例SQL 权重最高 |
| **表探索** | 用户想了解某张表 | 表结构 > 字段描述 > 业务术语 > 血缘关系 | 字段描述 权重最高 |
| **血缘分析** | 用户查上下游关系 | 血缘关系 > 表结构（上下游表）> 业务规则 | 血缘 权重最高 |
| **数据质量** | 用户关心数据可信度 | 质量指标 > 值分布 > 时效性信息 | 质量指标 权重最高 |
| **口径确认** | 用户问"GMV怎么算" | 业务术语 > 业务规则 > 示例SQL > 相关字段 | 业务术语 权重最高 |

#### 3.3.2 意图识别方式

| 方式 | 说明 | 适用场景 |
|------|------|---------|
| **消费方显式声明** | 请求中带 `intent: "text2sql"` | MCP 调用时，消费方知道自己要什么 |
| **关键词规则** | 从用户 query 中匹配关键词 | 简单场景，快速兜底 |
| **LLM 意图分类** | 用轻量模型对 query 做意图分类 | 复杂、模糊的场景 |

实际使用中三种方式**级联**：先看有没有显式声明 → 没有则走关键词规则 → 仍不确定则用 LLM 分类。

### 3.4 候选召回（Retrieval）

从上下文存储中找到可能相关的上下文单元。

#### 3.4.1 三路召回

单一检索方式不够，需要**三路并行召回**再合并：

```
用户 Query
    │
    ├─→ [关键词检索] 精确匹配表名、字段名、FQN
    │       ↓
    │   候选集 A（精确命中）
    │
    ├─→ [语义检索] Query Embedding → 向量相似度搜索
    │       ↓
    │   候选集 B（语义相关）
    │
    └─→ [图检索] 基于实体关系图的关联扩展
            ↓
        候选集 C（关联实体）
            │
            ▼
        合并去重 → 候选集
```

**三路各自的作用：**

| 检索方式 | 擅长 | 不擅长 | 示例 |
|---------|------|--------|------|
| **关键词检索** | 精确匹配名称、缩写、专有名词 | 同义词、语义理解 | 用户说"dws_order_summary_nd" → 精确命中 |
| **语义检索** | 理解意图、同义表达、模糊描述 | 精确名称匹配 | 用户说"订单汇总数据" → 语义匹配到订单汇总表 |
| **图检索** | 关联扩展、上下游发现 | 无关联关系时无效 | 命中订单表 → 自动扩展关联的支付方式维表 |

#### 3.4.2 图检索详解

图检索是区别于普通 RAG 的关键能力。基于实体关系图做关联扩展：

```
用户提到 dws_order_summary_nd
    │
    │ 图检索自动扩展
    ▼
┌─ dwd_order_detail_di（上游明细表，常用 JOIN）
├─ dim_pay_type（关联维表，支付方式）
├─ dim_user（关联维表，用户信息）
└─ GMV（关联业务术语）
```

**扩展规则：**

| 关系类型 | 扩展策略 | 扩展深度 |
|---------|---------|---------|
| 血缘关系 | 上游 1 层 + 下游 1 层 | 默认 1 层，可配置 |
| JOIN 关系 | 常用 JOIN 表全部召回 | 不限层 |
| 业务术语关联 | 表/字段关联的 Glossary 术语 | 直接关联 |
| 同域表 | 同 schema 下的高关联表 | 按关联度 top 3 |

#### 3.4.3 召回数量控制

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 关键词检索 top-K | 20 | 精确命中的候选数 |
| 语义检索 top-K | 30 | 语义相关的候选数 |
| 图扩展 max | 15 | 图检索扩展的最大实体数 |
| 合并后上限 | 50 | 去重合并后进入排序阶段的最大候选数 |

### 3.5 相关性排序（Ranking）

候选召回后，需要对所有候选上下文单元打分排序，决定哪些最终进入上下文包。

#### 3.5.1 排序因子

| 因子 | 权重 | 说明 |
|------|------|------|
| **召回得分** | 30% | 关键词匹配度 / 向量相似度 / 图关联强度 |
| **意图匹配度** | 25% | 该上下文类型与当前意图的配方匹配度（见 3.3.1） |
| **信息密度** | 20% | 单位 Token 内的有效信息量（避免冗长低效的上下文） |
| **新鲜度** | 15% | 越新的上下文分越高（避免过期信息误导 LLM） |
| **使用频率** | 10% | 历史上被使用且效果好的上下文加分 |

#### 3.5.2 排序策略

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| **加权线性** | 各因子按权重加权求和 | P0 阶段，简单有效 |
| **LLM Reranker** | 用轻量模型对候选集做精排 | P1 阶段，处理复杂 query |
| **场景专用** | 不同意图使用不同的权重配置 | 精细化运营阶段 |

### 3.6 Token 预算管理（Budget Management）

**这是组装层最实际的工程约束。** LLM 上下文窗口有限，不能无限塞。

#### 3.6.1 预算分配模型

一次请求的 Token 预算分配：

```
总 Token 预算（如 128K）
    │
    ├── System Prompt（固定）        ～ 2K tokens
    ├── 对话历史（动态）              ～ 变化
    ├── 上下文包（本模块控制）         ← 剩余空间
    └── 生成预留（留给输出）           ～ 4K tokens
    
上下文包可用 = 总预算 - System Prompt - 对话历史 - 生成预留
```

#### 3.6.2 上下文包内部的预算分配

上下文包内部也有优先级分配：

| 槽位 | 占比 | 内容 | 说明 |
|------|------|------|------|
| **核心上下文** | 50% | 直接命中的表结构、字段描述 | 必须保留，不可压缩 |
| **增强上下文** | 30% | 示例 SQL、业务术语、JOIN 路径 | 高价值补充，可按优先级裁剪 |
| **扩展上下文** | 20% | 关联表结构、血缘、文档片段 | 锦上添花，预算不够时优先砍 |

#### 3.6.3 预算不足时的降级策略

当上下文总量超出预算时，按以下优先级依次裁剪：

```
[1] 砍扩展上下文 — 关联实体只保留最相关的 top 3
        │
        │ 还不够？
        ▼
[2] 压缩增强上下文 — 示例 SQL 只保留最相似的 2 条，术语只保留直接相关的
        │
        │ 还不够？
        ▼
[3] 精简核心上下文 — 表结构只保留核心字段（主键+维度+度量+分区），隐藏非关键字段
        │
        │ 还不够？
        ▼
[4] 摘要兜底 — 将整段上下文压缩为一段自然语言摘要
```

## 模块 ④ 上下文交付（Context Delivery via MCP）

### 4.1 模块定位

> 通过标准化协议将组装好的上下文包交付给各种 AI 消费方，让任何 Agent 都能即插即用。

交付层是上下文管理对外的"门面"。组装层做得再好，如果交付协议不标准、不灵活，消费方接入成本高，产品价值就打了折扣。

**核心选型：MCP（Model Context Protocol）。**

### 4.2 为什么选 MCP

| 维度 | MCP | 自建 REST API |
|------|-----|--------------|
| **标准化** | Anthropic 主推的开放协议，生态在快速成长 | 自定义，每个消费方要单独对接 |
| **能力抽象** | 原生支持 Resources / Tools / Prompts 三种交付形式 | 需要自己设计接口结构 |
| **双向通信** | 支持服务端主动推送（如 Schema 变更通知） | 需额外建 WebSocket 或轮询 |
| **Agent 友好** | LLM Agent 原生理解 MCP 协议，可自主选择需要的上下文 | Agent 需要硬编码调用逻辑 |
| **生态兼容** | 兼容 Claude、LangChain、各类 Agent 框架 | 只能自己用 |

> MCP 不排斥 REST API。上下文管理可以同时暴露 MCP Server 和 REST API，MCP 面向 AI Agent，REST API 面向传统应用（如数据地图前端）。

### 4.3 MCP 三种交付原语

MCP 协议定义了三种核心原语（Primitives），上下文管理需要基于这三种原语设计交付能力：

```
┌─────────────────────────────────────────────────┐
│                MCP Server                        │
│                                                  │
│   ┌─────────────┐                                │
│   │  Resources   │  → 被动提供：消费方按需拉取     │
│   │  (资源)       │     表结构、术语表、文档片段    │
│   └─────────────┘                                │
│                                                  │
│   ┌─────────────┐                                │
│   │    Tools     │  → 主动执行：消费方调用动作     │
│   │  (工具)       │     搜索表、获取血缘、组装上下文 │
│   └─────────────┘                                │
│                                                  │
│   ┌─────────────┐                                │
│   │   Prompts    │  → 预组装模板：含场景化上下文    │
│   │ (提示词模板)   │     Text2SQL prompt、探索 prompt │
│   └─────────────┘                                │
└─────────────────────────────────────────────────┘
```

#### 4.3.1 Resources（资源）

**定义：** 静态或准静态的上下文数据，消费方可按 URI 拉取。

**适用场景：** 消费方明确知道需要哪些资源，按需获取。

| Resource | URI 格式 | 返回内容 | 示例 |
|----------|---------|---------|------|
| 表结构 | `context://tables/{fqn}/schema` | 增强后的表结构（含样本值、分类标签） | `context://tables/dm_trade.dws_order_summary_nd/schema` |
| 字段描述 | `context://tables/{fqn}/columns` | 所有字段的业务描述 | `context://tables/dm_trade.dws_order_summary_nd/columns` |
| 血缘关系 | `context://tables/{fqn}/lineage` | 上下游血缘链路 | `context://tables/dm_trade.dws_order_summary_nd/lineage` |
| 业务术语 | `context://glossary/{term}` | 术语定义、关联字段、计算规则 | `context://glossary/GMV` |
| 示例 SQL | `context://examples/{domain}` | 某领域的验证 SQL 对集合 | `context://examples/trade` |
| 表列表 | `context://tables?schema={schema}` | 某 schema 下所有表概要 | `context://tables?schema=dm_trade` |

**Resource 特性：**

| 特性 | 说明 |
|------|------|
| **URI 寻址** | 每个资源有唯一 URI，消费方可精确请求 |
| **MIME 类型** | 支持 `text/plain`（适合直接塞 prompt）和 `application/json`（适合程序处理） |
| **订阅变更** | 消费方可订阅某个 Resource，当内容变化时收到通知 |
| **缓存友好** | 每个 Resource 带版本号和 ETag，支持客户端缓存 |

#### 4.3.2 Tools（工具）

**定义：** 可执行的动作，消费方（或 LLM Agent）调用后返回结果。

**适用场景：** 需要动态计算、检索或组装的操作。Agent 可自主决定何时调用哪个 Tool。

| Tool | 输入参数 | 返回内容 | 说明 |
|------|---------|---------|------|
| `search_tables` | `query: string, top_k: number` | 匹配的表列表（FQN + 中文名 + 摘要） | 语义搜索表资产 |
| `get_table_context` | `fqn: string, intent: string` | 单表的完整上下文包 | 按意图组装单表上下文 |
| `assemble_context` | `query: string, intent: string, budget: number` | 组装好的上下文包 | **核心 Tool**：调用模块③的完整组装流程 |
| `get_lineage` | `fqn: string, depth: number, direction: "up"\|"down"\|"both"` | 血缘链路 | 获取指定深度的血缘关系 |
| `validate_sql` | `sql: string` | 语法校验结果 + 涉及的表/字段 | 校验 SQL 是否合法 |
| `get_join_path` | `table_a: string, table_b: string` | 推荐的 JOIN 路径和条件 | 两表之间怎么 JOIN |
| `search_glossary` | `query: string` | 匹配的业务术语列表 | 搜索业务口径定义 |
| `get_similar_queries` | `query: string, top_k: number` | 相似的历史 NL→SQL 对 | 查找类似的示例 SQL |

**最核心的 Tool 是 `assemble_context`**，它封装了模块③的全部能力：

```
消费方调用:
assemble_context({
  query: "近7天GMV趋势",
  intent: "text2sql",
  budget: 8000
})

返回:
{
  context_pack: { core: [...], enhanced: [...], extended: [...] },
  token_used: 7200,
  metadata: { recall_count: 35, packed_count: 4 }
}
```

**Tool 的关键设计原则：**

| 原则 | 说明 |
|------|------|
| **原子性** | 每个 Tool 做一件事，可组合使用 |
| **自描述** | Tool 的 name + description 要让 LLM 能自主判断何时调用 |
| **幂等性** | 相同参数多次调用结果一致 |
| **错误友好** | 返回结构化错误信息，LLM 能理解并调整策略 |

#### 4.3.3 Prompts（提示词模板）

**定义：** 预组装的 Prompt 模板，包含场景化的上下文和指令，消费方可直接使用或基于参数动态生成。

**适用场景：** 针对高频场景提供"开箱即用"的 Prompt，减少消费方的 Prompt 工程工作量。

| Prompt | 参数 | 内容 | 场景 |
|--------|------|------|------|
| `text2sql` | `query: string, tables?: string[]` | System Prompt + 表结构 + 术语 + 示例 SQL + 用户 Query | 自然语言转 SQL |
| `table_explorer` | `fqn: string` | 表全貌上下文 + 引导式问答指令 | 表探索对话 |
| `lineage_analyst` | `fqn: string, direction: string` | 血缘上下文 + 影响分析指令 | 血缘影响分析 |
| `glossary_qa` | `term: string` | 术语定义 + 关联规则 + 示例 | 口径答疑 |

**Prompt 模板示例（text2sql）：**

```
你是一个 SQL 生成助手。根据以下数据库上下文，将用户的自然语言问题转为 SQL。

## 可用表结构
{auto_assembled: 根据 query 自动组装的表结构}

## 业务术语
{auto_assembled: 相关业务术语}

## 参考示例
{auto_assembled: 相似的 NL→SQL 对}

## 规则
- 只使用上方列出的表和字段
- 时间分区字段 dt 格式为 yyyyMMdd
- 默认查最近一个分区，除非用户指定时间
- 金额字段单位为"分"，展示时需除以 100

## 用户问题
{user_query}
```

**Prompts 与 Tools 的区别：**

| 维度 | Prompts | Tools |
|------|---------|-------|
| **交付物** | 完整的 Prompt（可直接发给 LLM） | 数据（需要消费方自己组装 Prompt） |
| **灵活性** | 低（模板化） | 高（消费方自由组合） |
| **接入成本** | 极低（拿来就用） | 中等（需要编排调用逻辑） |
| **适用方** | 快速接入的消费方、轻量 Agent | 有自己 Prompt 策略的高级消费方 |

### 4.4 MCP Server 架构

```
┌──────────────────────────────────────────────┐
│              MCP Server（上下文管理）            │
│                                              │
│  ┌─────────────────────────────────────────┐ │
│  │           Capability Negotiation         │ │
│  │         （能力协商 — 握手阶段）              │ │
│  └─────────────────────────────────────────┘ │
│                                              │
│  ┌────────────┐ ┌──────────┐ ┌───────────┐  │
│  │ Resources  │ │  Tools   │ │  Prompts  │  │
│  │  Handler   │ │ Handler  │ │  Handler  │  │
│  └─────┬──────┘ └────┬─────┘ └─────┬─────┘  │
│        │             │             │         │
│        └─────────────┼─────────────┘         │
│                      │                       │
│              ┌───────▼───────┐               │
│              │  组装层 (③)    │               │
│              │  加工层 (②)    │               │
│              │  接入层 (①)    │               │
│              └───────────────┘               │
│                                              │
│  ┌─────────────────────────────────────────┐ │
│  │        Transport Layer（传输层）           │ │
│  │   Streamable HTTP  |  stdio（本地调试）    │ │
│  └─────────────────────────────────────────┘ │
└──────────────────────────────────────────────┘
```

#### 4.4.1 能力协商（Capability Negotiation）

MCP 连接建立时，Server 和 Client 先协商支持的能力：

```json
// Server → Client: 声明支持的能力
{
  "capabilities": {
    "resources": {
      "subscribe": true,
      "listChanged": true
    },
    "tools": {},
    "prompts": {
      "listChanged": true
    }
  }
}
```

消费方据此知道：
- 有哪些 Resources 可以拉
- 有哪些 Tools 可以调
- 有哪些 Prompts 可以用
- 是否支持订阅变更通知

#### 4.4.2 传输方式

| 传输方式 | 适用场景 | 说明 |
|---------|---------|------|
| **Streamable HTTP** | 生产环境，远程调用 | 基于 HTTP 的流式传输，支持跨网络部署 |
| **stdio** | 本地开发、调试 | 标准输入输出，MCP Inspector 等工具可直接调试 |

**生产环境推荐 Streamable HTTP**，支持：
- 认证鉴权（Bearer Token / API Key）
- 多租户隔离
- 负载均衡
- 请求限流

### 4.5 变更通知（Notifications）

当上下文发生变化时，MCP Server 主动通知已订阅的消费方：

| 事件 | 触发条件 | 通知内容 |
|------|---------|---------|
| **Schema 变更** | 表/字段新增、删除、类型变更 | 变更的表 FQN、变更类型、变更详情 |
| **术语更新** | Glossary 术语定义修改 | 变更的术语、新旧定义对比 |
| **资源列表变化** | 新增/删除了 Resource | `notifications/resources/list_changed` |
| **上下文质量告警** | 某实体的上下文质量分低于阈值 | 实体 FQN、质量分、缺失项 |

**通知机制：**

```
上下文管理（接入层检测到变更）
    │
    ▼
加工层重新加工受影响的上下文单元
    │
    ▼
MCP Server 发送 notification 给已订阅的 Client
    │
    ├─→ DataAgent: 下次组装时使用新上下文
    ├─→ 数据地图 Agent: 更新缓存的表信息
    └─→ 其他消费方: ...
```

### 4.6 认证与多租户

#### 4.6.1 认证方式

| 方式 | 适用场景 | 说明 |
|------|---------|------|
| **API Key** | 服务间调用 | 每个消费方分配独立 API Key |
| **Bearer Token** | 用户级请求 | 继承消费方的用户 Token，用于权限控制 |
| **mTLS** | 高安全要求 | 双向证书认证 |

#### 4.6.2 多租户隔离

不同消费方可能有不同的上下文权限：

```
DataAgent（数据分析师角色）
    → 可见: 所有表结构、字段描述、示例 SQL
    → 不可见: 数据质量内部评分、治理规则详情

数据地图 Agent（业务用户角色）
    → 可见: 表概要、字段描述、业务术语
    → 不可见: 原始 DDL、技术细节

管理后台（管理员角色）
    → 可见: 全部上下文 + 加工日志 + 质量报告
```

权限控制在 MCP Server 层做，消费方无感知——同一个 Tool 调用，不同角色返回不同范围的上下文。

### 4.7 REST API 兼容层

MCP 面向 AI Agent，但传统应用（如数据地图前端）可能不走 MCP。上下文管理同时提供 REST API：

| REST 接口 | 对应的 MCP 能力 | 消费方 |
|-----------|----------------|--------|
| `GET /api/v1/context/tables/{fqn}` | Resource: `context://tables/{fqn}/schema` | 数据地图前端 |
| `POST /api/v1/context/search` | Tool: `search_tables` | 搜索服务 |
| `POST /api/v1/context/assemble` | Tool: `assemble_context` | 不支持 MCP 的旧系统 |
| `GET /api/v1/context/glossary/{term}` | Resource: `context://glossary/{term}` | 业务系统 |

**REST API 和 MCP 共享同一套组装逻辑**，不是两套实现。REST API 只是 MCP 的一个 HTTP 适配层。

### 4.8 与现有 DataAgent 的对接方案

当前 DataAgent 的请求：

```json
// 现在
{
  "sessionId": "session_2026_xxx",
  "content": "近7天GMV趋势",
  "mentionTables": ["dm_trade.dws_order_summary_nd"]
}
```

接入上下文管理后的请求链路：

```
用户输入 "近7天GMV趋势"
    │
    ▼
DataAgent 前端（提取 mentionTables）
    │
    ▼
DataAgent 后端
    │
    │  调用 MCP Tool: assemble_context
    │  {
    │    query: "近7天GMV趋势",
    │    intent: "text2sql",
    │    mentioned_tables: ["dm_trade.dws_order_summary_nd"],
    │    budget: 8000
    │  }
    │
    ▼
上下文管理 MCP Server
    │
    │  执行组装流程：召回 → 排序 → 裁剪 → 组装
    │
    ▼
返回上下文包
    │
    ▼
DataAgent 后端（拼装最终 Prompt = System + 上下文包 + 对话历史 + 用户 Query）
    │
    ▼
LLM（生成 SQL / 回答）
    │
    ▼
DataAgent 前端（SSE 流式展示）
```

**改动范围：**

| 组件 | 改动 | 大小 |
|------|------|------|
| DataAgent 前端 | 无改动（仍然传 content + mentionTables） | 无 |
| DataAgent 后端 | 新增 MCP Client，在调 LLM 之前先请求上下文管理获取上下文包 | 中等 |
| 上下文管理 | 新建 MCP Server（本产品） | 新建 |

## 五、模块 ⑤ — 上下文质量 & 治理

### 5.1 模块定位

> **如果不知道上下文"够不够好"，就无法系统性地提升 Text2SQL 准确率。**

模块 ③ 负责"组装"，模块 ⑤ 负责"度量"。质量 & 治理解决三个问题：

1. **覆盖度** — 有多少资产已经有可用的上下文？哪些还是空白？
2. **质量评分** — 已有的上下文质量如何？够不够好？
3. **治理管控** — 谁能看什么上下文？操作痕迹能不能追溯？

### 5.2 上下文覆盖度看板

#### 5.2.1 为什么需要

上下文管理接入了元数据、文档、业务口径等多种来源，但不代表每张表、每个字段都有完整的上下文。如果一张核心业务表缺少字段描述和业务口径，Text2SQL 对这张表的查询准确率就会显著下降。

覆盖度看板的目的：**让产品和数仓团队一眼看到"哪些资产还是裸奔状态"，驱动补全。**

#### 5.2.2 覆盖度维度

| 维度 | 统计口径 | 计算方式 |
|------|---------|---------|
| **表描述覆盖率** | 有 description 的表 / 总表数 | `count(description IS NOT NULL) / count(*)` |
| **字段描述覆盖率** | 有 description 的字段 / 总字段数 | 同上，按字段维度 |
| **业务口径覆盖率** | 关联了 Glossary Term 的字段 / 总字段数 | 通过 OpenMetadata Glossary 关联关系统计 |
| **血缘覆盖率** | 有血缘关系的表 / 总表数 | 存在上游或下游的表占比 |
| **示例 SQL 覆盖率** | 关联了示例 SQL 的表 / 总表数 | 上下文加工后存在 `type=example_sql` 的 Context Unit |
| **向量化覆盖率** | 已生成 embedding 的 Context Unit / 总 Context Unit 数 | 加工模块处理进度 |

### 5.3 上下文质量评分

#### 5.3.1 评分模型

每个 Context Unit 独立评分，汇聚到表级和 Schema 级。采用 **四维评分模型**：

| 维度 | 权重 | 评分逻辑 | 分值范围 |
|------|------|---------|---------|
| **完整度 (Completeness)** | 35% | 必填字段的填充率（description、type、tags 等） | 0-100 |
| **新鲜度 (Freshness)** | 25% | 距上次同步/更新的时间衰减 | 0-100 |
| **可信度 (Trustworthiness)** | 25% | 来源权威性 × 人工审核状态 | 0-100 |
| **有效度 (Effectiveness)** | 15% | 下游使用反馈（SQL 执行成功率、用户纠正率） | 0-100 |

**综合评分** = Σ（维度分 × 权重）

#### 5.3.2 各维度评分细则

**完整度评分：**

| Context Unit 类型 | 必填字段 | 评分规则 |
|-------------------|---------|---------|
| `table_schema` | description, columns[].description, columns[].type | 每个必填字段缺失扣 20 分 |
| `column_detail` | description, data_type, business_meaning | 每缺一项扣 25 分 |
| `lineage` | upstream_tables, downstream_tables, transformation | 缺 transformation 扣 30 分 |
| `glossary` | term, definition, related_tables | 缺 definition 扣 50 分 |
| `example_sql` | sql, description, scenario | 缺 scenario 扣 20 分 |

**新鲜度评分：**

```
score = max(0, 100 - decay_rate × days_since_update)
```

| 来源类型 | 衰减速率 (decay_rate) | 说明 |
|---------|----------------------|------|
| 元数据（Schema） | 2 分/天 | Schema 变更频率中等，50 天归零 |
| 血缘关系 | 1 分/天 | 血缘相对稳定，100 天归零 |
| 业务口径 | 0.5 分/天 | 口径变更较少，200 天归零 |
| 示例 SQL | 3 分/天 | SQL 易过时，约 33 天归零 |

**可信度评分：**

| 因素 | 分值 |
|------|------|
| 来源：OpenMetadata API 自动采集 | 基础分 60 |
| 来源：人工录入 / 文档提取 | 基础分 40 |
| 经过人工审核确认 | +30 |
| 有多个来源交叉验证 | +10 |
| 被标记为"待核实" | -20 |

**有效度评分：**

| 信号 | 影响 |
|------|------|
| 基于该上下文生成的 SQL 执行成功 | +5（单次，累积上限） |
| 用户对结果点了"有用" | +10 |
| 用户手动纠正了 SQL | -15 |
| 用户反馈"上下文有误" | -30 |

#### 5.3.3 质量评分数据结构

**Context Unit 级别：**

```json

  "context_unit_id": "cu_tbl_dm_trade_dws_order_001",
  "quality_score": {
```

#### 5.3.4 质量评分计算策略

| 策略 | 说明 |
|------|------|
| **定时批量计算** | 每天凌晨全量重算一次所有 Context Unit 的质量评分 |
| **事件触发增量更新** | Context Unit 被更新、收到反馈信号时，实时重算该 Unit 的评分 |
| **评分缓存** | 质量评分写入独立表，查询时直接读取，不实时计算 |

### 5.4 上下文治理

#### 5.4.1 访问控制

上下文管理面向多个 AI 消费方，需要控制"谁能拿到什么上下文"。

**权限模型：**

```
消费方（Consumer）
  └── 角色（Role）
        └── 权限集（Permission Set）
              ├── 可访问的 Schema 范围
              ├── 可访问的 Context Unit 类型
              └── 可使用的 Tool 列表
```

| 角色 | 说明 | 权限范围示例 |
|------|------|-------------|
| `full_access` | 完全访问 | 所有 Schema、所有类型、所有 Tool |
| `schema_scoped` | Schema 级别隔离 | 仅 `dm_trade.*` 和 `dm_user.*` |
| `read_only` | 只读，无 Tool 调用权 | 仅 Resources 读取，不能调用 assemble_context |
| `sandbox` | 沙箱模式 | 仅脱敏后的上下文，不含真实数据预览 |

**权限检查时机：**

```
消费方请求 → 身份认证（API Key / OAuth）
  → 角色解析
  → 请求的 Schema / Context Unit 类型是否在权限范围内？
    → 是：正常返回
    → 否：返回 403 + 提示缺少哪个权限
```

#### 5.4.2 数据脱敏规则

某些上下文可能包含敏感信息（如示例 SQL 中的真实数据、字段描述中的业务规则细节）。

| 脱敏级别 | 规则 | 适用场景 |
|---------|------|---------|
| **L0 无脱敏** | 原样返回 | 内部 DataAgent，已授权 |
| **L1 值脱敏** | 示例 SQL 中的常量值替换为占位符 | 外部消费方 |
| **L2 结构脱敏** | 隐藏敏感字段的 description，仅保留字段名和类型 | 低权限角色 |
| **L3 不可见** | 整个 Context Unit 不返回 | 无权限 |

**脱敏配置示例：**

```json
{
  "rule_id": "mask_pii_fields",
  "match": {
    "tags": ["PII", "敏感字段"],
    "context_unit_type": ["column_detail", "example_sql"]
  },
  "action": "L2",
  "description": "包含 PII 标签的字段描述对外部消费方隐藏业务含义"
}
```

#### 5.4.3 审计日志

每一次上下文交付都应留痕，用于安全审计和问题排查。

**审计日志结构：**

```json
{
  "audit_id": "aud_20260411_083000_001",
  "timestamp": "2026-04-11T08:30:00Z",
  "consumer": {

      "query": "dm_trade.dws_order_summary_nd 的字段含义",
      "intent": "field_detail",
      "mention_tables": ["dm_trade.dws_order_summary_nd"]
    }
  },

```

**审计日志用途：**

| 场景 | 使用方式 |
|------|---------|
| **安全审计** | 定期检查是否有异常访问模式（如高频拉取全量上下文） |
| **问题排查** | Text2SQL 生成错误时，回溯当时给了什么上下文，定位是上下文问题还是 LLM 问题 |
| **使用统计** | 聚合审计日志，生成模块 ⑥ 需要的使用分析数据 |
| **合规报告** | 证明敏感数据的访问均经过权限控制和脱敏处理 |

**审计日志保留策略：**

| 级别 | 保留时长 | 说明 |
|------|---------|------|
| 摘要日志（无 unit_ids 详情） | 180 天 | 用于统计分析 |
| 完整日志（含返回的上下文快照） | 30 天 | 用于问题排查，存储成本较高 |
| 安全事件日志（403/异常访问） | 365 天 | 合规要求 |

### 5.5 质量治理工作流

覆盖度和质量评分的最终目的是**驱动改善**。质量治理不是看看数字就完了，需要闭环：

```
发现问题 → 生成任务 → 分配责任人 → 完成补全 → 重新评分
```

#### 5.5.1 自动生成治理任务

基于质量评分和覆盖度缺口，系统自动生成治理任务：

| 触发条件 | 生成任务 | 优先级 |
|---------|---------|--------|
| 高频查询表（30 天 > 50 次）且质量评分 < 60 | "补全 {表名} 的字段描述和业务口径" | P0 |
| Schema 级别字段描述覆盖率 < 50% | "推进 {schema} 的字段描述补全" | P1 |
| Context Unit 新鲜度评分 = 0（超期未更新） | "检查 {表名} 的元数据是否仍然有效" | P1 |
| 有效度评分连续 7 天 < 40 | "核查 {表名} 上下文准确性，纠正率过高" | P0 |

#### 5.5.2 治理任务数据结构

```json
{
  "task_id": "gov_task_20260411_001",
  "type": "completeness_gap",
  "target": {
    "fqn": "dm_trade.dwd_order_detail_di",

  },
  "priority": "P0",
  "reason": "近 30 天被查询 87 次，但质量评分仅 45",

  "status": "open",                   // open → in_progress → done → verified

  "due_date": "2026-04-18"
}
```
