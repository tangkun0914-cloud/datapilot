# 元数据 & 上下文管理 — 产品定位与架构总览

> 版本：v0.1 | 日期：2026-04-10 | 状态：草稿

---

## 一、背景

### 1.1 业务背景

团队当前在做 **Text2SQL** 方向，核心产品是 **DataAgent**（数据地图 Agent）。Text2SQL 的核心难题不是 SQL 生成本身，而是 **让 LLM 拿到足够好的上下文** —— 表叫什么、字段含义、表间关系、业务口径、数据质量如何。上下文质量直接决定 SQL 生成的准确率。

### 1.2 当前问题

DataAgent 目前传给后端的上下文非常有限：

```json
{
  "sessionId": "session_2026_1712766000000",
  "content": "看下 @dm_trade.dws_order_summary_nd 的字段",
  "mentionTables": ["dm_trade.dws_order_summary_nd"]
}
```

- **content**：用户原始输入文本
- **mentionTables**：从用户输入中提取的表 FQN（通过 `@` 提及）

**没有任何上下文加工层。** 后端拿到的只有表名，缺少表结构、字段含义、血缘关系、业务口径等关键信息。这意味着 LLM 需要自己去"猜"或者后端自己再去查，无法保证上下文的质量和一致性。

### 1.3 核心诉求

需要搭建完整的基础设施，让 DataAgent / Text2SQL 能获取到高质量、结构化的上下文，从而提升 SQL 生成准确率和用户体验。

## 二、产品定位

### 2.1 两个独立产品，不是一个

| 产品 | 面向用户 | 定位 |
|------|---------|------|
| **元数据管理** | 数仓团队 | 底层基础设施，采集、存储、治理原始元数据 |
| **上下文管理** | AI 消费方（DataAgent、Text2SQL 等） | 独立产品，从多种来源加工、组装、交付上下文 |

**关键决策：上下文管理是独立产品，不是元数据管理的子模块。**

原因：上下文管理的数据来源不只是元数据，未来会接入文档、知识库、业务规则、示例 SQL 等多种来源。如果框在元数据管理里面，产品边界会被锁死。

### 2.2 产品关系

```
                    数据来源（多种）
          ┌──────────┼──────────────────┐
          │          │                  │
  ┌───────▼──────┐   │          ┌───────▼───────┐
  │  元数据管理   │   │          │  文档/知识库   │
  │ (OpenMetadata)│   │          │  业务规则      │
  │  面向数仓     │   │          │  示例 SQL      │
  └───────┬──────┘   │          └───────┬───────┘
          │          │                  │
          │    是其中一个数据源           │
          │          │                  │
          ▼          ▼                  ▼
  ┌─────────────────────────────────────────────┐
  │            上下文管理（独立产品）               │
  │                                             │
  │  接入 → 加工 → 组装 → 交付（MCP）             │
  └──────────────────┬──────────────────────────┘
                     │ MCP 对外提供上下文服务
          ┌──────────┼──────────────┐
          ▼          ▼              ▼
     数据地图     DataAgent      未来其他
      Agent      Text2SQL       AI 场景
```

### 2.3 数据地图的位置

**数据地图是元数据管理的上层应用，面向业务用户。** 它和 DataAgent 一样，都是元数据的消费方。

```
元数据管理（底层，面向数仓）
    ├── 数据地图（上层应用，面向业务用户，看/搜/理解数据资产）
    └── 上下文管理（独立产品，面向 AI 消费方）
            └── 数据地图 Agent（基于上下文管理的外延应用）
```

数据地图 Agent 不直接读元数据，而是通过上下文管理获取加工后的上下文。

## 三、元数据管理 — 基于 OpenMetadata

### 3.1 为什么选 OpenMetadata

| 需求 | OpenMetadata 能力 |
|------|-------------------|
| 表结构、字段元数据 | 原生支持，自动采集 |
| 表间关系、血缘 | 列级血缘（Column-level Lineage） |
| 业务语义 / 口径 | Glossary（业务术语表）+ Tag 分类体系 |
| 数据质量信号 | 内置 Data Profiler + Data Quality 框架 |
| 数仓连接器 | Hive、Spark、Doris、ClickHouse 等 |
| API 对外服务 | API-first 架构，所有元数据可通过 REST API 获取 |

### 3.2 元数据管理的核心能力（概要）

在 OpenMetadata 之上封装，提供以下能力：

| 能力 | 说明 |
|------|------|
| **元数据采集** | 连接器管理、定时/增量采集、Schema 变更检测 |
| **元数据存储** | 统一元数据模型，存储表/字段/血缘/标签 |
| **业务语义管理** | Glossary 术语表建设、字段业务含义标注 |
| **数据质量** | Profiling、质量规则、质量评分 |
| **数据血缘** | 表级 + 列级血缘自动解析与展示 |
| **API 服务** | RESTful API 对外提供元数据查询能力 |

### 3.3 需要注意的点

1. **Glossary 要认真建设**：OpenMetadata 只是框架，业务语义需要产品 + 业务推动填充。Text2SQL 效果很大程度取决于字段业务描述质量。
2. **采集频率**：OpenMetadata 是定时 Ingestion，不是实时同步。需评估数仓变更频率，确定合理的采集周期。
3. **上下文管理通过 OpenMetadata API 获取元数据**，不直接访问数仓底层。

## 四、上下文管理 — 产品能力全景

### 4.1 产品定义

> **上下文管理是一个通用的上下文工厂，能接入多种知识来源，按场景加工，并通过 MCP 协议交付给各种 AI 消费方。**

### 4.2 六大能力模块

```
┌─────────────────────────────────────────────────────────┐
│                    上下文管理产品                          │
│                                                         │
│  ① 上下文接入 → ② 上下文加工 → ③ 上下文组装 → ④ 上下文交付  │
│                                                         │
│  ⑤ 上下文质量 & 治理          ⑥ 上下文反馈 & 生命周期       │
└─────────────────────────────────────────────────────────┘
```

| 模块 | 核心职责 | 一句话 |
|------|---------|--------|
| **① 上下文接入** | 多来源统一收入 | 把各种"原料"收进来 |
| **② 上下文加工** | 标准化、向量化、增强 | 原料变半成品 |
| **③ 上下文组装** | 按场景动态拼装 | **核心差异化** — 精准投喂，不是一股脑塞 |
| **④ 上下文交付** | MCP 协议对外服务 | 标准接口，即插即用 |
| **⑤ 上下文质量 & 治理** | 覆盖度、质量评分、权限 | 知道"够不够好" |
| **⑥ 上下文反馈 & 生命周期** | 效果反馈、版本管理 | 越用越好的飞轮 |

> 各模块的详细产品设计见第 ② 篇文档：《上下文管理-产品能力详细设计》

## 五、当前现状分析

### 5.1 DataAgent 现有上下文传递

| 字段 | 内容 | 来源 |
|------|------|------|
| `sessionId` | 会话 ID（`session_2026_{timestamp}`） | 前端生成 |
| `content` | 用户原始输入，保留 `@` 前缀 | 用户输入 |
| `mentionTables` | 去重后的表 FQN 数组 | `extractMentionFqnsFromText()` 从用户输入提取 |

**请求示例：**
```
POST /api/v1/agent/chat/stream
{
  "sessionId": "session_2026_1712766000000",
  "content": "看下 @dm_trade.dws_order_summary_nd 的字段，@dm_trade.dwd_order_detail_di 的血缘",
  "mentionTables": ["dm_trade.dws_order_summary_nd", "dm_trade.dwd_order_detail_di"]
}
```

### 5.2 FQN 提取机制

- 正则：`/@([\w][\w.]*\.\w[\w.]*)/g`
- 匹配 `@schema.table` 或 `@schema.db.table` 格式
- 自动去重，保持出现顺序
- 代码位置：`src/utils/extractMentionFqns.js`

### 5.3 响应中返回的元数据

后端通过 SSE 流式返回 `tableDetail`，但**只包含 3 个字段**：

```json
{
  "fqn": "dm_trade.dws_order_summary_nd",
  "cnName": "订单汇总表",
  "owner": "张三(zhangsan)"
}
```

**明确不包含 `description`**（PRD 约束），表描述需要通过数据地图资产详情页查看。

### 5.4 支持的意图类型

| 类型 | 意图 | 关键词 |
|------|------|--------|
| Type 1 | 表搜索列表 | 订单、找、相关、列表、风控 |
| Type 2 | 单表详情（默认） | 明确指向一张表 |
| Type 3 | 字段列表 | 字段、schema、表结构 |
| Type 4 | 血缘摘要 | 血缘、上游、下游 |
| Type 5 | 数据预览 | 预览、抽样 |
| Type 6 | 生产调度 | 生产、调度、执行 |
| Type 7 | 脚本信息 | DDL、建表语句、查看脚本 |
| Type 8 | SELECT 查询 | SELECT、查询语句 |

### 5.5 GAP 分析

| 维度 | 现状 | 目标 |
|------|------|------|
| 上下文内容 | 仅表 FQN | 表结构 + 字段含义 + 血缘 + 业务口径 + 示例 SQL |
| 上下文来源 | 用户手动 `@` 提及 | 自动识别 + 多来源融合 |
| 上下文加工 | 无 | 标准化、向量化、Schema 增强 |
| 上下文组装 | 无 | 按意图动态组装，Token 预算管理 |
| 上下文交付 | 直接传 FQN 字符串 | MCP 协议标准化交付 |
| 上下文质量 | 无评估 | 覆盖度看板、质量评分、反馈闭环 |

## 六、术语表

| 术语 | 含义 |
|------|------|
| **FQN** | Fully Qualified Name，全限定名，格式如 `schema.table` |
| **MCP** | Model Context Protocol，模型上下文协议，Anthropic 提出的标准化上下文交付协议 |
| **Context Factory** | 上下文工厂，即本文档中"上下文管理"产品的内部代号 |
| **Text2SQL** | 自然语言转 SQL，用户用自然语言提问，系统自动生成 SQL 查询 |
| **DataAgent** | 数据 Agent，基于 LLM 的数据交互助手 |
| **Glossary** | 业务术语表，OpenMetadata 中管理业务语义的功能模块 |
| **Embedding** | 向量嵌入，将文本转为向量表示，用于语义检索 |
| **Token Budget** | LLM 上下文窗口的 Token 容量限制 |
| **SSE** | Server-Sent Events，服务端推送事件，当前 DataAgent 使用的流式响应协议 |
