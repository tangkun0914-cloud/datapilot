## 三、前后端接口契约（重点）

### 3.1 接口一览

| # | 接口 | 方法 | 用途 | 前端调用时机 |
|---|------|------|------|-------------|
| 1 | `/api/monitoring/impact/topology` | GET | 获取拓扑初始数据（仅根节点） | 打开战情室时 |
| 2 | `/api/monitoring/impact/children` | GET | 获取节点子节点（逐层展开） | 用户点击 `+` 时 |
| 3 | `/api/monitoring/impact/core-path` | GET | 获取核心链路拓扑 | 开启「仅看核心链路」开关时 |
| 4 | `/api/monitoring/impact/summary` | GET | 获取影响统计摘要（含全局清单） | 打开战情室时 |
| 5 | `/api/monitoring/impact/ai-analysis` | GET | 获取 AI 分析结果 | 轮询（3s × 10 次） |
| 6 | `/api/monitoring/impact/create-group` | POST | 一键拉群 | 用户确认创建时 |
| 7 | `/api/monitoring/impact/snapshot` | GET | 获取历史快照 | snapshot 模式打开时 |

> **刷新逻辑**：页面右上角【刷新】按钮并行重新调用接口 2（对已展开节点）+ 接口 4，无需独立刷新接口。
> **恢复通知**：告警 resolve 时后端自动检查并推送企微群消息，前端无需额外调用。

### 3.2 接口详细定义

#### 接口 1：获取拓扑初始数据

```
GET /api/monitoring/impact/topology?eventId={eventId}
```

**响应体**：
```json
{
  "alertSource": "quality",          // "dev" | "integration" | "quality"
  "currentNode": {                   // TaskNode，画布根节点
    "id": "task_12345",
    "taskName": "dwd_fund_df_repayment",
    "taskType": "SparkSQL",
    "owner": "马殿林(madianlin)",
    "impactStatus": "failed",        // 见下方枚举
    "isCurrentNode": true,
    "hasChildren": true,             // 控制 +/− 控件
    "isCore": false,
    "isPolluted": false,
    "isRootCause": false,            // 仅 DQC 告警为 true
    "slaRiskType": null,
    "startTime": "2026-04-02 08:23",
    "endTime": "2026-04-02 08:25",
    "depth": 0,
    "statusColor": null,             // 可选：覆盖默认颜色
    "statusText": null               // 可选：覆盖默认文案
  }
}
```

#### 接口 2：获取节点子节点

```
GET /api/monitoring/impact/children?nodeId={nodeId}&eventId={eventId}
```

**响应体**：
```json
{
  "nodes": [{ /* TaskNode */ }],
  "edges": [{ "source": "task_12345", "target": "task_67890", "edgeType": "dependency" }]
}
```


**响应体**：同接口 2 格式（nodes + edges），仅包含根→核心任务的关键路径节点。

```
GET /api/monitoring/impact/ai-analysis?eventId={eventId}
```

**响应体**：
```json
{
  "status": "ready",                 // "loading" | "ready" | "failed"
  "impactJudgment": "受影响下游实例 56 个，其中核心实例 3 个...",
```

#### 接口 6：一键拉群

```
POST /api/monitoring/impact/create-group
```

**请求体**：
```json
{
  "eventId": "E-10000641",
  "groupName": "【故障应急】ERROR dwd_fund_df_repayment 运行失败",
  "members": ["zhangsan", "lisi", "madianlin"]
```

```json

  "alreadyExists": false,

```

```

```

```json

  "topology": { "nodes": [], "edges": [] },
  "summary": { /* 同接口 4 格式 */ }
}
```

> **后端要求**：告警产生时预计算，状态变为 resolved/falsePositive 时自动保存快照，`exists` 恒为 `true`。

## 四、核心数据模型

### 4.1 TaskNode（拓扑节点）

```json
{
  "id": "task_12345",
  "taskName": "dwd_fund_df_repayment",
  "taskType": "SparkSQL",
  "owner": "马殿林(madianlin)",
  "impactStatus": "failed",
  "isPolluted": false,
  "isRootCause": false,
  "isCurrentNode": true,
  "nodeType": "task",
  "hasChildren": true,
  "isCore": false,
  "slaRiskType": null,
  "startTime": "2026-04-02 08:23",
  "endTime": "2026-04-02 08:25",
  "depth": 0,
  "statusColor": null,
  "statusText": null,
  "errorSummary": "Task [dwd_fund...] Failure Warning"
}
```

### 4.2 impactStatus 枚举与五色映射

| impactStatus | 颜色 | 色值 | 卡片文案 | 图例归属 |
|-------------|------|------|---------|---------|
| `success` | 🟢 绿 | `#52C41A` | 成功 | 执行成功 |
| `failed` | 🔴 红 | `#FF4D4F` | 已失败 | 失败/停止 |
| `timeout` | 🔴 红 | `#FF4D4F` | 已超时 | 失败/停止 |
| `stopped` | 🔴 红 | `#FF4D4F` | 已停止 | 失败/停止 |
| `dqc_threshold` | 🔴 红 | `#FF4D4F` | 阈值异常 | 失败/停止 |
| `running` | 🟠 橙 | `#FA8C16` | 运行中 | 运行/排队 |
| `waiting` | 🟠 橙 | `#FA8C16` | 依赖等待 | 运行/排队 |
| `delayed` | 🟠 橙 | `#FA8C16` | 排队中 | 运行/排队 |
| `other` | 🔵 蓝 | `#1890FF` | 串行等待 | 其他状态 |
| `pending` | ⚪ 灰 | `#8c8c8c` | 未运行 | 未生成 |
| `not_generated` | ⚪ 灰 | `#8c8c8c` | 未生成 | 未生成 |

### 4.3 附加标签字段（不影响主状态颜色）

| 字段 | 类型 | 视觉 | 使用场景 |
|------|------|------|---------|
| `isPolluted` | boolean | 紫色底部标签「☣ 可能污染」 | 仅 DQC WARN 下游，递归传递 |
| `isCore` | boolean | 蓝色底部标签「核心任务」 | 有 SLA 配置的任务 |
| `slaRiskType` | string/null | 红/橙色标签 | `dual_breached` / `start_breached` / `end_breached` |
| `isRootCause` | boolean | 🎯 前缀 | 仅 DQC 告警的当前质量任务节点 |

## 五、告警来源与影响逻辑对照

| 告警来源 | alertSource 值 | 拓扑数据源 | 方向 | DQC 根因高亮 | 下游附加标识 |
|---------|---------------|-----------|------|-------------|------------|
| 数据开发 | `dev` | DolphinScheduler 调度依赖 | 仅下游 | ❌ 不做 | 展示实际运行状态 |
| 数据集成 | `integration` | DolphinScheduler 调度依赖 | 仅下游 | ❌ 不做 | 展示实际运行状态 |
| 数据质量 ERROR | `quality` | DQC 表→调度任务→调度依赖 | 下游 | ✅ 🎯 高亮 | 阻断（下游实际为依赖等待） |
| 数据质量 WARN | `quality` | DQC 表→调度任务→调度依赖 | 下游 | ✅ 🎯 高亮 | ☣ 污染风险（递归传递） |

**DQC 两类监控事件必须区分**：

| 监控事件 | 含义 | 根节点 impactStatus |
|---------|------|-------------------|
| 质量监控 - 运行失败 | 规则/质检作业执行过程中报错 | `failed` |
| 质量监控 - 触发异常阈值 | 执行成功但产出数据越过阈值 | `dqc_threshold` |

> **禁止**用 `failed` 表达「触发异常阈值」，两者语义完全不同。

## 八、后端待对接 Checklist

以下为前端当前依赖 Mock 的部分，需后端逐一落实：

- [ ] **接口 1** `GET /topology`：返回根节点 + `alertSource` + `hasChildren`
- [ ] **接口 2** `GET /children`：按 nodeId 返回直接子节点及连线
- [ ] **接口 3** `GET /core-path`：返回根→核心任务的关键路径（SLA 任务）
- [ ] **接口 4** `GET /summary`：全量下游统计 + ownerMatrix + **slaItems**（注意：字段名为 `slaItems` 非 `slaBreaches`，需包含所有配置 SLA 的任务含可控状态，`slaBreachType` 枚举含 `controllable`）+ affectedTaskInstances
- [ ] **接口 5** `GET /ai-analysis`：异步预计算结果，status 字段必须准确（loading/ready/failed）
- [ ] **接口 6** `POST /create-group`：对接企微建群 API，含 eventId 去重逻辑；报告卡片内容由后端根据 eventId 自行组装，前端仅传 eventId/groupName/members
- [ ] **接口 7** `GET /snapshot`：告警终态时保存快照，确保 `exists` 恒为 true
- [ ] **恢复通知**：告警状态变更为 `resolved` 时，后端在现有 resolve 流程中追加一步：检查该 `eventId` 是否曾创建过应急群，若有则自动向群内推送恢复通知消息卡片。根据 `resolveType`（`manual` 人工确认 / `auto` 系统自动恢复）选择对应消息模板（详见 PRD 6.8 节）。前端无需额外调用
- [ ] **`alertSource` 字段**：响应中必须携带，供前端切换 DQC 高亮/污染等逻辑
- [ ] **`impactStatus` 枚举对齐**：11 个状态值必须与前端五色映射表一致
- [ ] **DQC 运行失败 vs 触发异常阈值**：根节点必须区分返回 `failed` 和 `dqc_threshold`
- [ ] **DQC 组告警（规则合并告警）**：同一表/任务下多个 DQC 规则同批次触发时，拓扑根节点保持单一（不分裂）；多规则明细由后端拼接到日志内容中（供 Tab 4 展示）；AI Prompt 需注入触发的规则数量与明细
