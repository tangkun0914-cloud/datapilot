# DataOps 智能影响评估与协同平台 - V1.0 PRD

## 一、 文档概述与产品定位

### 1.1 产品定位
基于“数据血缘”与“大模型代码语义分析”的端到端数据运维管控平台。以“影响链路可视化”为基础，串联“变更评审、自动熔断、智能协同”，解决 DataOps 场景下“出了问题/发生变更时，不知道影响有多大、找人难、沟通慢、误报多”的核心痛点，实现从“事后补救”向“事前预防与智能降噪”的演进。

### 1.2 核心业务目标
*   **看清影响（可视化）**：直观界定异常/变更任务的“爆炸半径”，量化对下游资产和人员的影响。
*   **看准影响（智能降噪）**：引入 LLM 大模型对代码进行语义级比对，精准过滤无影响的下游任务，降低告警疲劳。
*   **快速止血（闭环管控）**：提供质量熔断、一键管控、一键拉群等抓手，打破跨团队沟通壁垒，缩短故障恢复时间（MTTR）。

## 二、 目标用户与使用场景

### 2.1 目标用户
*   **数据开发工程师**：任务报错或主动变更代码时，需要评估对下游的影响，并通知相关方。
*   **数据运维工程师 (SRE)**：全局监控任务运行状态，处理高危告警，进行级联重跑或阻断操作。
*   **数据消费者（业务方/分析师）**：作为下游最终报表/数据的负责人，需要及时获知上游故障或变更情况，评估业务影响。

### 2.2 核心用户故事 (User Stories)
*   **场景 1：任务报错/延迟（故障响应）**
    *   *作为数据运维*，当发现核心任务 A 失败时，我希望一键查看其下游受影响的链路，并自动拉起包含所有受影响负责人的应急群，以便快速对齐信息并执行“一键冻结下游”操作。
*   **场景 2：表结构/逻辑变更（变更评审）**
    *   *作为数据开发*，当我修改了表 B 的字段或计算口径时，我希望大模型能帮我自动扫描下游所有依赖该表的代码，告诉我哪些任务会报错（高危），哪些不受影响（安全），并一键向受影响的高危任务负责人发起变更评审。
*   **场景 3：数据质量异常（自动熔断）**
    *   *作为数据平台管理员*，我希望当某张核心表产出的数据空值率飙升时，系统能自动阻断下游依赖任务的执行，防止脏数据污染最终的 CEO 看板。

## 三、 核心功能详细说明

### 模块一：智能影响评估引擎 (Intelligent Assessment Engine)

#### 1.1 基础链路追踪与可视化
*   **多层级拓扑展示**：基于当前异常/变更节点，向下游展开依赖关系。默认展示 **3 层**，支持节点折叠/展开下钻。
*   **状态实时映射**：在画布节点上通过颜色/图标直观区分状态（如：🔴失败、🟡延迟、🔵变更中、⚪未运行、🟢成功）。
*   **节点详情透出**：悬浮或点击节点，可查看节点基础信息（负责人、任务类型、最近一次运行时长、报错摘要）。

#### 1.2 核心链路一键透视
*   **核心资产过滤**：提供“仅看核心链路”的全局拨动开关。
*   **过滤逻辑**：开启后，系统自动隐藏非核心的边缘任务及中间临时表，仅高亮展示带有“SLA 保障”、“高管看板”、“核心资产”等标签的下游节点及最终产出物，帮助用户在海量节点中聚焦重点。

#### 1.3 AI 代码语义级评估分级 (LLM-based Impact Triage) 🌟
*   **触发时机**：当节点发生代码变更、表结构（Schema）变更时触发。
*   **分析逻辑**：系统提取上游变更内容（如：删除了字段 X，修改了枚举值 Y），并拉取下游所有相关节点的源代码（SQL/Python），提交给大模型进行语义对比。
*   **智能分级输出**：
    *   **高危/致命 (High/Critical)**：上游变更直接破坏了下游逻辑（如删除了下游强依赖的字段）。AI 需输出具体报错的行号和风险解释。
    *   **中危 (Medium)**：逻辑变更可能导致下游数据口径变化（如金额计算公式改变），但不一定会报错。
    *   **低危/无影响 (Low/None)**：上游变更的字段下游并未引用。AI 判定为安全，实现“智能降噪”。

### 模块二：多维影响面报告 (Multi-dimensional Impact Report)

#### 2.1 资产与人员双维统计面板
*   **资产维度**：统计受影响的节点总数、核心节点数、最终受影响的业务报表/API 数量。
*   **人员维度（责任矩阵）**：汇总受影响的下游负责人名单，及每人名下受牵连的任务数（例：张三 5个，李四 2个）。支持点击人名反向高亮拓扑图中的对应节点。

#### 2.2 SLA 破线预估 (SLA Prediction)
*   **延迟推演**：针对“延迟”状态的节点，系统基于历史平均运行时长（Avg. Duration），自动推演下游链路的预计完成时间。
*   **破线预警**：如果推演出的完成时间晚于下游核心资产承诺的 SLA 时间，在报告中进行醒目预警（如：“预计下游【财务核心报表】将晚于 09:00 产出，存在破线风险”）。

#### 2.3 动态报告生成与分享
*   **一键生成**：将拓扑截图、AI 诊断结论、资产/人员统计数据整合为一份结构化的《影响评估报告》。
*   **分享与导出**：支持生成只读 Web 链接分享，或导出为 PDF/Markdown 格式。

### 模块三：闭环管控与智能协同 (Closed-loop Control & Collaboration)

#### 3.1 异常自动熔断与手动管控
*   **质量异常自动熔断 (Auto-Circuit Breaker)**：
    *   **规则联动**：与数据质量中心（DQC）联动。
    *   **阻断机制**：当检测到当前节点产出数据存在严重质量问题（触发强规则，如主键重复、空值飙升）时，系统**自动挂起（Suspend）**下游所有依赖任务的执行，防止脏数据蔓延。
*   **手动干预矩阵 (Action Bar)**：
    *   **针对失败**：提供“一键重跑（当前及下游）”、“一键置成功（强制放行）”、“一键挂起（冻结下游）”。
    *   **针对延迟**：提供“一键提权/插队”（提升调度优先级）。

#### 3.2 变更评审工作流 (Change Review Workflow)
*   **发起评审**：变更发起人可基于 AI 评估报告，一键向受影响（特别是 AI 判定为中高危）的下游负责人发起“变更评审”。
*   **反馈闭环**：下游负责人收到系统通知后，可在平台内进行“同意变更”或“驳回并留言”的操作。
*   **审计记录**：所有的评审意见和操作记录留存，作为变更发布的卡点依据（如：必须所有高危下游同意后，变更才能发布到生产环境）。

#### 3.3 一键拉群与报告推送 (One-click War Room)
*   **智能建群**：系统自动读取“责任矩阵”中的人员名单，调用企业通讯工具（企微/飞书/钉钉）API，一键创建“故障应急群”或“变更讨论群”。
*   **上下文注入**：建群成功后，机器人自动将《影响评估报告》卡片及 AI 诊断结论推送至群内，确保所有人进群即掌握全局上下文，避免重复沟通。

## 四、 页面与交互结构建议

### 4.1 核心页面：影响评估工作台 (Impact Assessment Workspace)
采用**左右分栏（或上下分栏）**的沉浸式布局：
*   **左侧/上方：可视化画布区 (Canvas)**
    *   承载 DAG 拓扑图，提供缩放、平移、节点折叠/展开功能。
    *   顶部提供工具栏：【仅看核心链路】开关、【层级深度】选择器。
*   **右侧/下方：评估报告与操作区 (Panel)**
    *   **Tab 1 - 统计与报告**：展示资产/人员统计数字、SLA 预估、AI 诊断结论。
    *   **Tab 2 - 变更评审/协同**：展示评审进度、留言记录。
    *   **底部固定操作条 (Action Bar)**：放置【一键拉群】、【发起评审】、【一键挂起】、【级联重跑】等核心操作按钮。

## 五、 关键指标与验收标准 (Success Metrics)

1.  **故障响应效率**：从发现异常到通知所有下游负责人的时间，从平均的 N 分钟缩短至 **1 分钟内**（通过一键拉群）。
2.  **变更误报率**：通过 AI 代码语义分析，将传统血缘变更告警的误报率降低 **50% 以上**。
3.  **脏数据阻断率**：核心链路的质量异常自动熔断成功率达到 **100%**，实现 0 脏数据穿透。
4.  **系统性能**：拓扑图默认 3 层展开的渲染响应时间不超过 **2 秒**。
