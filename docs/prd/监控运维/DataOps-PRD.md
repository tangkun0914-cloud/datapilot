# DataOps 监控运维中心 - PRD

## 1. 核心目标与产品定位
**产品定位**：DataOps 监控运维中心是 DataPilot 平台的核心运维枢纽，旨在将离散的任务告警、数据质量告警进行统一管理，并通过“数据血缘”和“任务依赖”实现影响评估与智能协同，最终实现 DataOps 的自动化治愈。

**核心目标**：
1. **统一告警管理**：将数据开发、数据集成、数据质量的告警统一收口，标准化展示。
2. **智能影响评估**：基于血缘和调度依赖，自动计算告警的“爆炸半径”，精准定位根因和受影响的下游资产。
3. **高效协同与闭环**：通过 ChatOps 一键拉群、自动同步状态，并提供“阻断下游”、“级联重跑”等一键式运维操作。

## 2. 问题定义
*   **告警孤岛**：开发任务失败、质量规则异常分散在不同模块，运维人员难以全局掌控。
*   **排查成本高**：告警发生时，人工排查上下游影响范围耗时耗力，容易遗漏重要业务报表。
*   **协同效率低**：故障处理依赖人工拉群、口头同步，缺乏基于事件的自动化应急响应机制。
*   **恢复链路长**：根因修复后，需要手动逐个恢复下游任务，容易出错且效率低下。

## 3. 核心功能模块与用户故事
### 3.1 统一告警列表 (Alerts)
*   **用户故事**：作为运维人员，我希望在一个列表中看到所有的告警事件（包括任务失败、质量异常），并能通过高级筛选快速定位问题。
*   **核心功能**：
    *   统一展示告警卡片（标题、级别 ERROR/WARN、状态、来源、监控事件等）。
    *   支持强/弱质量规则的聚合展示（单规则单事件 vs 弱规则打包事件）。
    *   高级筛选：数据表名搜索、来源筛选、监控事件联动、任务/表负责人独立筛选。

### 3.2 影响评估与智能协同 (Impact & ChatOps)
*   **用户故事**：作为数据Owner，当我的核心表出现质量问题时，我希望系统能自动告诉我上游是哪个任务导致的，以及下游有哪些报表受到了影响，并能一键拉群解决。
*   **核心功能**：
    *   **可视化拓扑图**：展示当前告警节点的上游链路（疑似根因）和下游链路（受影响节点）。
    *   **影响清单**：列出受影响的任务、表、API、BI报表及责任人。
    *   **一键拉群**：提取相关责任人，一键创建企微/飞书/钉钉群，并注入告警上下文卡片。

### 3.3 运维控制与级联回溯 (Ops Control)
*   **用户故事**：作为运维人员，我希望在发现严重质量问题时能一键阻断下游，并在修复后一键重跑所有受影响的任务。
*   **核心功能**：
    *   **阻断下游**：一键挂起受影响的下游任务，防止脏数据蔓延。
    *   **级联重跑**：根因修复后，基于拓扑路径一键恢复执行。

### 3.4 通知策略管理 (Policies)
*   **用户故事**：作为项目管理员，我希望能在测试或重跑数据时，临时关闭某些通知策略，避免无效告警打扰。
*   **核心功能**：
    *   通知策略列表支持“状态”开关（开启/关闭）。
    *   关闭策略时需二次确认防误操作，关闭后仅停止发送通知，不影响告警事件生成。

## 4. 页面结构与信息架构
*   `/ops/dashboard`：监控大盘（全链路监控视图与核心指标）
*   `/ops/alerts`：告警列表（统一告警卡片、高级筛选）
*   `/ops/alerts/:id`：告警详情页（包含基础信息、日志片段、**「影响评估」Tab**）
*   `/ops/policies`：通知策略管理（策略列表、状态开关）

## 5. MVP 范围界定 (Phase 1)
*   **实现**：统一告警列表页（支持质量告警接入与聚合展示）、通知策略状态开关、告警详情页的基础信息与「影响评估」可视化拓扑图（Mock数据）、一键拉群交互（前端模拟）。
*   **暂不实现**：真实的调度系统API打通（阻断下游/级联重跑在 Phase 2 实现）、全自动 DataOps 治愈（Phase 3）。
