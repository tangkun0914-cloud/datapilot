# 数据质量影响评估 — 阻断策略矩阵

## 前置规则说明

1. **ERROR 运行失败不阻断**：规则运行失败可能由集群资源等基础设施原因导致，若因此大批阻断下游不合理，因此 ERROR + 运行失败 不执行阻断。
2. **ERROR 触发异常阈值阻断**：数据确认超出阈值，属于已确认的数据质量问题，执行阻断。
3. **WARN 永不阻断**：弱规则属于警告级别，下游正常执行，仅做风险提示。
4. **混合场景继承规则**：WARN 继承同表所有活跃 ERROR 事件中的**最严格阻断状态**。只要存在任意一条 ERROR 为"阻断"，WARN 即继承"阻断"。

## 三、混合场景 — 从 WARN 进入

> 同一张表同时存在 ERROR 和 WARN 告警，用户从 WARN 告警进入影响评估。
> 核心原则：WARN 继承同表活跃 ERROR 事件的最严格阻断状态。

| 序号 | 告警级别 | 同表 ERROR 子类型 | 根节点展示 | 是否阻断下游 | 下游附加标识 | 交叉提示 |
|------|---------|-----------------|-----------|------------|------------|---------|
| 7 | ERROR + WARN，从 WARN 进 | ERROR 全部为**运行失败** | 告警态（橙色），标记 WARN | **不阻断**（继承 ERROR 真实状态：不阻断） | 紫色"可能污染"标签 | 顶部黄色 Banner："该表同时存在 ERROR 级别告警（运行失败），但未阻断下游。[查看 ERROR 告警 →]" |

## 四、阻断继承判定逻辑（伪代码）

```
function resolveBlockingStatus(currentAlert, allActiveAlerts) {
  // 1. 获取同表所有活跃 ERROR 事件
  const sameTableErrors = allActiveAlerts.filter(a =>
    a.tableName === currentAlert.tableName
    && a.severity === 'ERROR'
    && a.status === 'firing'
  )

  // 2. 当前告警是 ERROR，直接返回自身阻断状态
  if (currentAlert.severity === 'ERROR') {
    return {
      blocked: currentAlert.monitorEvent === '触发异常阈值',
      label: currentAlert.monitorEvent === '触发异常阈值' ? '阻断' : null,
      crossBanner: null
    }
  }

  // 3. 当前告警是 WARN，继承同表 ERROR 的最严格状态
  const hasBlockingError = sameTableErrors.some(
    e => e.monitorEvent === '触发异常阈值'
  )
  const hasAnyError = sameTableErrors.length > 0

  if (hasBlockingError) {
    return {
      blocked: true,
      label: '阻断',
      crossBanner: {
        type: 'error',
        message: `该表同时存在 ERROR 级别告警，下游已被阻断。`,
        link: sameTableErrors.find(e => e.monitorEvent === '触发异常阈值').id
      }
    }
  }

  if (hasAnyError) {
    // 存在 ERROR 但全部是运行失败（不阻断）
    return {
      blocked: false,
      label: '可能污染',
      crossBanner: {
        type: 'warning',
        message: `该表同时存在 ERROR 级别告警（运行失败），但未阻断下游。`,
        link: sameTableErrors[0].id
      }
    }
  }

  // 4. 纯 WARN，无 ERROR
  return {
    blocked: false,
    label: currentAlert.monitorEvent === '触发异常阈值' ? '可能污染' : null,
    crossBanner: null
  }
}
```

## 五、决策流程图

```
用户点击告警卡片 → 进入影响评估 Drawer
    │
    ├─ 当前告警是 ERROR？
    │   ├─ 运行失败 → 不阻断，无下游标识
    │   └─ 触发异常阈值 → 阻断，下游标"阻断"
    │   └─ 同表有 WARN？→ 底部小字提示（P2）
    │
    └─ 当前告警是 WARN？
        ├─ 同表无活跃 ERROR → 不阻断
        │   ├─ 运行失败 → 无标识
        │   └─ 触发异常阈值 → 下游标"可能污染"
        │
        └─ 同表有活跃 ERROR → 查询 ERROR 子类型
            ├─ 任一 ERROR 为"触发异常阈值"
            │   → 继承阻断，下游标"阻断"
            │   → 顶部红色 Banner + 跳转链接
            │
            └─ 全部 ERROR 为"运行失败"
                → 不阻断，下游标"可能污染"
                → 顶部黄色 Banner + 跳转链接
```

## 六、Banner UI 规格

### 红色 Banner（阻断继承）
> 触发条件：从 WARN 进入，同表存在"触发异常阈值"类型的 ERROR
> 文案：`该表同时存在 ERROR 级别告警（E-{eventId}），下游任务已被阻断。` [查看 ERROR 告警 →]
> 样式：红色背景，位于 Drawer 内容区顶部

### 黄色 Banner（存在 ERROR 但未阻断）
> 触发条件：从 WARN 进入，同表存在 ERROR 但全部为"运行失败"
> 文案：`该表同时存在 ERROR 级别告警（运行失败），但未阻断下游。` [查看 ERROR 告警 →]
> 样式：黄色背景，位于 Drawer 内容区顶部

### 底部小字（从 ERROR 进入时的 WARN 提示）
> 触发条件：从 ERROR 进入，同表存在 WARN
> 文案：`该表同时存在 WARN 级别告警（弱规则异常 N 条）`
> 样式：灰色小字，位于 Drawer 底部，优先级 P2
