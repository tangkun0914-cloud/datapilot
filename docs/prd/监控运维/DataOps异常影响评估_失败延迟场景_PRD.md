# DataOps 异常影响评估与协同中心 - V1.0 PRD (聚焦失败/延迟场景)

## 一、 文档概述与产品定位

### 1.1 产品定位：中立的底层基础服务
「影响评估」是一个**中立的、底层的数据资产与任务影响面分析服务（Service）**。
它通过解析数据血缘与调度依赖关系，为调用方提供精准的影响面量化与全链路可视化能力。
它**不是**监控运维模块的专属功能，也**没有**单一固定的系统菜单入口。它的产品形态类似于“数据地图”，是**无处不在**的——它可以被无缝嵌入到告警排查、任务发布、资产浏览等任何需要评估影响面的业务流程中。

### 1.2 核心业务目标（解耦评估与处置）
*   **纯粹的量化与可视化（核心能力）**：清晰界定目标对象（如报错任务、待变更代码）的影响范围，量化受波及的下游资产数量、核心任务以及相关负责人。
*   **服务化输出（I/O 契约）**：作为中立服务，仅输出客观的“评估结果”（拓扑图、统计矩阵、AI 影响总结）。至于评估后的“处置动作”（如一键拉群、挂起重跑、阻断发布等），由具体的**调用场景**（如监控运维、发布审批）自行决定和实现。

---

## 二、 目标用户与调用方

影响评估服务的受众不再局限于“人”，而是扩展到了“Agent”。

### 2.1 用户
*   **数据开发工程师**：
    *   **被动排障**：任务报错时，快速评估下游影响面，决定修复优先级。
    *   **主动变更**：修改代码或表结构发布前，预判变更可能导致的下游破坏。
*   **数据消费者（业务方/分析师）**：作为下游资产负责人，获知上游故障或变更带来的影响，调整业务预期。

### 2.2 Agent
*   **DataOps Sub-Agent**：当用户向 Agent 询问某个报错任务时，Agent 在后台静默调用该服务，获取该任务的下游波及范围和受牵连的核心资产名单，从而在回复用户时，能准确提示故障的严重程度并给出修复建议。
*   **CodeGeneration**：在自动生成或修改SQL代码后，调用影响评估服务，判断生成的代码若提交运行，会对下游产生多大的影响，以此作为代码安全性的校验环节。

## 三、 核心使用场景

影响评估服务覆盖所有“主动”与“被动”的探查需求：

### 3.1 被动响应场景（当前 V1.0 落地场景）
*   **监控告警排查**：当调度任务失败、超时、SLA 破线或 DQC 质量校验不通过时，触发影响评估，精准统计受牵连的资产和人员，并提供直观的上下游关系图，辅助快速止血。（**落地在监控运维模块**）

### 3.2 主动评估场景（未来扩展场景）
*   **任务变更与发布拦截**：开发人员修改了某张核心表的产出逻辑，发布上线前触发评估，若影响面过大则要求更高级别的审批。
*   **数据级联更新预判**：业务需要重跑某天的数据，提前评估重跑操作会级联触发多少下游任务，消耗多少计算资源。
*   **资源下线评估（降本增效）**：数据团队在清理无用表或废弃任务前，调用影响评估服务。若输出显示下游“0 实例、0 核心任务”，则可安全下线；若仍有下游依赖，则自动通知相关负责人确认，避免“拔出萝卜带出泥”。
*   **纯粹的链路探查**：用户在数据地图中浏览某张表时，单纯想知道“如果这张表出问题，下游谁会受影响”，无报错前提下的日常探查。

## 四、 核心服务能力模型 (I/O 契约)

作为中立服务，必须具备标准化的输入输出契约，以便与不同业务场景结合。不同上下文（Context）下的输入参数会有差异化，以支持后端做精准的范围推演：

### 4.1 输入契约 (Input)

| 场景 | Target Entity (评估目标) | Context (上下文) | 额外输入 (Extra Payload) |
| :--- | :--- | :--- | :--- |
| **告警排查** (当前落地) | 任务实例 ID | `alert` + `eventId` | 告警等级 (Error/Warn)、监控事件类型 |
| **变更发布** (未来扩展) | 任务 ID / 表名 | `change` | 变更类型（代码逻辑 / Schema / 调度配置） |
| **纯链路探查** (未来扩展) | 任务 ID / 表名 | `explore` | 无 |
| **Agent 调用** (未来扩展) | 任务 ID / 表名 | `agent` | 调用方 Agent 标识、拟执行的动作描述 |

### 4.2 输出契约 (Output)

*   `Topology Data`：受影响的 DAG 拓扑结构（节点与连线，包含节点的客观运行/调度状态）。
*   `Impact Scope`：量化范围数据（受影响总实例数、核心实例数、受牵连的负责人矩阵）。
*   `AI Summary`：**纯粹的影响面总结**（如：“当前异常导致下游 5 个核心任务处于依赖等待状态”）。
    *   *注：中立服务输出的 AI Summary **不包含**特定场景的处置建议。调用方（如监控运维、发布审批）在拿到该 Summary 后，需根据自身业务逻辑**叠加专属的处置建议**（例如：监控运维场景追加“建议立即拉群协同”；发布审批场景追加“影响面过大，建议提级至总监审批”）。*

## 五、 本次 V1.0 迭代范围与边界

为了确保服务能快速落地并验证价值，V1.0 版本明确以下边界：

### 5.1 落地场景边界与架构分层
*   **首发场景**：仅在**「监控运维」**模块中落地。作为中立服务的第一个“大客户”，监控运维将调用影响评估服务，并将其包装在“告警详情”与“告警列表”的交互入口中。
*   **后续举措分离**：V1.0 战情室中看到的「一键拉群」、「一键挂起」、「级联重跑」等按钮，**属于监控运维场景的专属举措**，不属于影响评估底层服务本身的能力。
*   **代码架构要求**：影响评估的核心能力（拓扑计算、统计、AI 总结）与监控运维的场景举措（拉群、挂起等）**在前后端代码架构上也必须保持严格分层**。核心组件应具备独立渲染和调用的能力，为后续其他场景（如发布变更、Agent 调用）的快速接入做好工程准备。

### 5.2 评估维度的边界（量化范围 vs 评估程度）
*   **先做“范围 (Scope)”**：V1.0 重点解决“影响了多少任务、影响了谁、核心任务有哪些”的问题。
*   **暂缓“程度 (Severity)”**：对于“影响到底有多严重（如轻微、一般、致命）”的定性评估目前难以标准化，需在后续版本中重新设计评估模型，V1.0 暂不涉及程度的绝对量化。

## 六、 核心功能详细说明 (V1.0 监控运维场景落地)

### 6.1 影响评估在监控运维中的能力边界（重要）
为避免产品与数据能力错配，影响评估在**两类告警**下的能力**不一致**，必须写死如下边界：

**A. 数据开发 / 数据集成类告警**（运行失败、运行超时、SLA 启动超时、SLA 完成超时）
*   **数据源**：拓扑图基于 **DolphinScheduler 调度依赖关系**，非表级数据血缘。
*   **只做下游影响评估**：以当前告警对应的**调度任务**为根，沿调度依赖向下游展开；**不回答**「是否由 DQC 导致」「上游 DQC 卡点是谁」——系统**不提供**从该类告警**向上游追溯 DQC 根因**的能力。
*   **展示粒度**：拓扑画布以**逻辑任务节点**为粒度（任务名、负责人等）；全局影响清单（Tab 3）展示下游**任务实例**（含实例 ID、调度批次等，以后端能力为准）。

**B. DQC（告警来源：数据质量）触发类告警**（DQC 运行失败、DQC 触发告警 Warn/Error 等）
*   **映射机制**：DQC 告警关联的数据表 → 找到该表对应的调度任务 → 以该调度任务为根，沿调度依赖关系展开下游影响。拓扑图数据源同样为 **DolphinScheduler 调度依赖**。
*   **展示粒度**：与 A 类一致——拓扑画布以逻辑任务节点为粒度，全局影响清单（Tab 3）展示下游任务实例。
*   **根因表达**：当前异常主体即为 DQC 任务本身；可在可视化中突出当前 DQC 节点，**不属于**「从开发任务反查 DQC」类追溯。

### 模块一：异常影响评估引擎（6.2 ~ 6.4）

> 涵盖拓扑展示、核心任务识别、异常影响判定规则三部分。

#### 6.2 基础链路追踪与可视化 (单节点辐射)
*   **单节点评估边界**：本次 V1.0 版本的评估范围**严格限定为“单一异常节点”**的下游影响评估。不支持同时选中多个异常节点进行合并评估。
*   **按需逐层拓扑展示**：进入战情室时**仅展示当前告警节点（根节点）**。对于包含未展示下游的节点，其右侧显示「+」控件，点击后**仅展开一层**直接子节点；展开后控件切换为「−」，点击「−」立即隐藏该节点延伸出的所有下游子图。严格遵循“逐层探索，拒绝全量渲染”原则。
*   **节点卡片信息结构**：
    *   **顶区**：任务实际运行状态（如：执行成功、失败/停止、运行/排队、其他状态、未生成），色系与 6.2.1 节状态映射表一致。若根节点本身为失败/停止状态，统一为红色展示。
    *   **主体**：任务名（过长省略，悬浮展示完整名称）、开始时间、结束时间（分两行展示）、负责人（格式：`姓名(邮箱前缀)`）。
    *   **标识标签**：根据节点属性展示”核心任务”标签及”SLA破线”类标签（如 SLA双重破线、SLA启动破线、SLA完成破线）。
*   **根因高亮（仅 DQC 告警）**：当**当前告警来源为数据质量（DQC）**时，可将**当前 DQC 任务节点**在拓扑中做醒目高亮（如 🎯 / 动效），表示「质量卡点即当前事件主体」。**数据开发/集成类告警**下**不做**「从当前任务向上游反查 DQC 根因」的高亮或链路展示。
*   **节点详情透出**：**悬浮**节点时，弹出 Tooltip 展示节点完整任务名（便于复制）；**点击**节点时，右侧面板自动跳转到 Tab 4（日志详情）展示该节点的日志内容（详见 7.2 节 Tab 4 说明）。

#### 6.2.1 节点状态与展示映射

画布节点展示的是 **DolphinScheduler 的实际运行状态**，通过颜色直观区分。状态色系与现有调度系统图例保持一致：

| 展示状态 | 颜色 | 对应的 DolphinScheduler 实际状态 |
|---------|------|-------------------------------|
| **执行成功** | 🟢 绿色 | `SUCCESS`, `FORCED_SUCCESS` |
| **失败/停止** | 🔴 红色 | `FAILURE`, `NEED_FAULT_TOLERANCE`, `KILL`, `STOP`, `READY_STOP` |
| **运行/排队** | 🟠 橙色 | `RUNNING_EXECUTION`, `DISPATCH`, `SUBMITTED_SUCCESS`, `DELAY_EXECUTION`, `WAITING_DEPEND`, `WAITING_THREAD` |
| **其他状态** | 🔵 蓝色 | `PAUSE`, `READY_PAUSE` |
| **未生成** | ⚪ 灰色 | `PENDING` |

**污染风险附加标识（仅 DQC WARN 场景）**：当告警来源为 DQC 且级别为 WARN 时，下游节点在保持实际状态颜色的基础上，额外在卡片底部叠加 ☣️ **污染风险**标识，表示虽然任务实际状态可能正常，但数据质量存疑。该标识**递归传递至所有后代节点**。

#### 6.2.2 边界场景：无下游依赖
当告警任务本身为叶子节点（不存在下游依赖）时，仍打开影响评估抽屉，画布仅展示当前告警节点自身，右侧统计数据全部为 0，AI 总结提示「当前任务无下游依赖，不存在下游影响。建议关注任务自身的报错修复。」

#### 6.3 核心任务识别与关键路径穿透 (Core Task Identification & Path Penetration)
针对底层任务（如 ODS/DWD 层）报错导致下游扇出极广、层级极深的”毛线球”视觉灾难，系统采用**”规则 + 图算法 + 大模型”**三层架构识别核心任务并实现关键路径穿透：

##### 核心任务判定逻辑（分层，优先级从高到低）

| 层级 | 判定方式 | 输入数据 | 数据来源 | 说明 |
|------|---------|---------|---------|------|
| **第 1 层：规则优先** | 有 SLA 配置的任务 → 直接标记为核心 | SLA 启动时间 / SLA 完成时间 | 调度系统（DolphinScheduler） | 不依赖 LLM，确定性最高 |
| **第 2 层：图算法候选** | 出度 Top N 的枢纽节点 + 叶子节点 → 进入候选集 | 节点出度（被依赖次数）、是否为叶子节点 | 图算法实时计算 | 筛选出结构上重要的节点 |
| **第 3 层：LLM 语义判断** | 对候选集做语义分析 → 输出是否为高优业务任务 | 任务名、产出表名 | 调度元数据 + 血缘元数据 | 识别含 `finance`、`trade`、`ceo_dashboard` 等语义的任务，打上 **[AI 预警: 高优业务]** 标签 |

> **V1.0 落地说明**：本期前端实际交互采用”**按需逐层展开（详见 6.2 节）+ 核心链路一键穿透**”方案（详见 7.2 节「仅看核心链路」工具栏说明）。第 1 层（SLA 规则）本期落地；第 2-3 层（图算法 + LLM 语义推断）作为产品愿景在后续版本迭代。

##### 边界场景：无核心任务
当下游链路中不存在任何配置 SLA 的任务时，开启「仅看核心链路」开关后，画布仅展示当前告警节点（根节点）自身，无路径可穿透。

#### 6.4 异常场景与影响判定规则 (Impact Assessment Rules)

画布上所有下游节点展示的是 **DolphinScheduler 的实际运行状态**（详见 6.2.1 节），系统不做额外的状态推演。本节定义不同告警类型下的**影响范围判定**与**特殊展示规则**：

**1. 数据开发/集成任务告警**（运行失败、运行超时、SLA 启动超时、SLA 完成超时）
*   **评估边界**：以当前告警对应的调度任务为根，沿调度依赖向下游展开，所有下游节点展示其在 DolphinScheduler 中的实际运行状态。
*   **能力边界**：仅评估下游影响；**不**判定、不展示「是否由 DQC 引起」，**不**在图中增加「上游 DQC 根因节点」链路。

**2. DQC（数据质量来源）任务告警：**

DQC 告警下，系统除展示下游节点的实际运行状态外，还需根据**告警级别 × 监控事件类型**两个维度判定阻断行为与附加标识，**不区分国内/印尼等区域**：

| 告警级别 | 监控事件 | 根节点展示 | 是否阻断下游 | 下游附加标识 |
|---------|---------|-----------|------------|------------|
| **ERROR** | 触发异常阈值 | 阈值告警态（红色），文案「触发异常阈值」——任务执行成功但结果超阈值，需与运行失败区分 | **阻断**（下游实际状态为依赖等待） | 红色「阻断」标签 |
| **ERROR** | 运行失败 | 失败态（红色），文案「运行失败」 | **不阻断**（下游按正常调度执行） | 无 |
| **WARN** | 触发监控阈值 | 阈值告警态（橙色），颜色标记 WARN | **不阻断**（下游按正常调度执行） | 紫色「可能污染」标签 |
| **WARN** | 运行失败 | 失败态（橙色），颜色标记 WARN | **不阻断**（下游按正常调度执行） | 无 |

*   **根节点区分「运行失败」与「触发异常阈值」**：两者均可能为红色/橙色系，但文案和图标必须区分——「运行失败」表示任务执行过程中报错，「触发异常阈值」表示任务执行成功但产出数据越过质量阈值。禁止用 `failed` 混淆两种语义。
*   **ERROR 触发异常阈值阻断**：数据确认超出阈值，属于已确认的数据质量问题，执行阻断，下游节点展示红色「阻断」标签。
*   **ERROR 运行失败不阻断**：规则运行失败可能由集群资源等基础设施原因导致，若因此大批阻断下游不合理，因此 ERROR + 运行失败 不执行阻断，下游无附加标识。
*   **WARN 永不阻断**：弱规则属于警告级别，下游正常执行。仅在触发异常阈值时，下游节点叠加紫色「可能污染」标签做风险提示。

**3. DQC 规则合并告警（组告警）展示规范**
当同一表/任务下的多个数据质量规则（如字段空值、枚举值异常等）在同一次调度中触发告警时，底层系统通常将其合并为一条告警事件下发。在影响评估中：
*   **拓扑节点保持单一**：拓扑图根节点仍代表该表/任务本身，不因规则数量而分裂节点。
*   **规则明细通过日志透出**：多规则明细由后端拼接到日志内容中，用户在 Tab 4（日志详情）中即可查看所有触发的具体规则（规则名、当前值、阈值等）。影响评估内**不新增**独立的规则明细表组件；完整规则明细亦可在告警详情抽屉中查看。
*   **AI 总结归因**：AI 影响总结需明确指出触发了多个规则（如”当前表触发了 2 个弱规则告警：字段空值率、枚举值异常”），辅助经办人快速判断。

### 模块二：多维影响面报告（6.5 ~ 6.6）

> 涵盖统计面板、SLA 破线判定两部分。

#### 6.5 资产与人员双维统计面板
统计数据基于**全量下游数据**（与 Tab 3 全局影响清单数据源一致），不受画布当前展开层级影响，打开影响评估时即展示完整影响面。
*   **资产维度**：统计受影响的实例总数。
*   **人员维度（受影响负责人）**：汇总受影响的下游负责人名单，及每人名下受牵连的实例数（例：张三(sanzhang) 5个，李四(sili) 2个）。当人数较多时，表格支持分页展示（默认每页 10 条）。

#### 6.6 SLA 监控状态 (SLA Monitoring Status)
*   **前提说明**：SLA 配置取决于用户在调度系统中的设置，不是每个任务都有 SLA 配置，且 SLA 启动时间和 SLA 完成时间可以独立配置（任务可能只配了其中一项）。仅对配置了 SLA 的下游任务进行状态监控与破线判定。
*   **判定逻辑**：系统将下游任务的**实际启动/完成状态**与其配置的 **SLA 承诺时间**进行实时比对，判定当前状态：
    *   **SLA 启动破线**：当前时间已超过配置的 SLA 启动时间，但任务尚未启动。
    *   **SLA 完成破线**：当前时间已超过配置的 SLA 完成时间，但任务尚未完成。
    *   **SLA 双重破线**：同时满足启动破线和完成破线条件。
    *   **可控 (未破线)**：当前时间未超过 SLA 承诺时间，或任务已在承诺时间内启动/完成。
*   **展示方式**：以卡片列表形式展示所有配置了 SLA 的任务及其时间对比。**破线任务与未破线（可控）任务均在列表中展示**，按风险等级（双重破线 > 单项破线 > 可控）降序排列，以便用户全面掌握 SLA 资产的安全水位。同时，顶部提供三个维度的筛选器：**任务名称搜索**（子串匹配）、**负责人多选过滤**、**SLA状态多选过滤**（双重破线、启动破线、完成破线、可控），帮助用户在海量任务中快速定位目标。

##### SLA 监控状态定义

| SLA 状态 | 判定条件 | 展示样式 |
|----------|---------|---------|
| **SLA 双重破线** | 启动时间和完成时间均已超过 SLA 承诺 | 红色高危标签 |
| **SLA 启动破线** | 仅启动时间已超过 SLA 承诺（完成时间在 SLA 内或未配置） | 橙色标签 |
| **SLA 完成破线** | 完成时间已超过 SLA 承诺（启动时间在 SLA 内或未配置） | 红色高危标签 |
| **可控** | 尚未触发任何破线规则，处于安全范围内 | 绿色安全标签 |

### 模块三：闭环管控与智能协同（6.7 ~ 6.8）

> 涵盖操作按钮矩阵、一键拉群与恢复通知两部分。

#### 6.7 异常手动管控矩阵 (Action Bar)
在影响评估视图的右侧面板底部固定展示操作按钮。

##### V1.0 按钮可用性矩阵

| 按钮 | V1.0 状态 | 说明 |
|------|-----------|------|
| **一键拉群** | **可用** | 调用**企业微信** API 创建应急群（详见 6.8 节），V1.0 仅对接企微 |
| **一键挂起** | 展示但置灰 | Hover 显示 Tooltip：「该功能将在后续版本支持」；待任务运维能力接入后启用 |
| **级联重跑** | 展示但置灰 | Hover 显示 Tooltip：「该功能将在后续版本支持」；待任务运维能力接入后启用 |
| **一键置成功** | 展示但置灰 | Hover 显示 Tooltip：「该功能将在后续版本支持」；待任务运维能力接入后启用 |

*   **针对”失败”节点**：

    *   **级联重跑 (Cascade Rerun)**：当前节点修复后，一键触发当前节点及所有受影响下游节点的重跑。

#### 6.8 一键拉群与报告推送 (One-click War Room)

##### 建群交互流程（V1.0）

##### 故障恢复通知（V1.0）

当告警状态变更为 `resolved` 时，系统检查该告警是否曾创建过应急群：
*   **已创建过应急群** → 自动向该群推送恢复通知消息（根据恢复方式区分模板）。
*   **未创建过应急群** → 不推送。

**恢复方式 1：人工确认恢复**（操作人手动点击「已解决」按钮）

```

告警任务 {任务名} 已被标记为「已解决」。
恢复方式：人工确认
操作人：{操作人姓名(邮箱前缀)}
操作时间：{解决时间}
请相关方确认下游任务运行状态。
```

**恢复方式 2：系统自动恢复**（系统检测到任务实例已完成，自动标记为已解决）

```
✅ 【告警状态变更通知】
告警任务 {任务名} 已被标记为「已解决」。
恢复方式：系统自动恢复（检测到任务实例已完成）

```

> **说明**：系统自动恢复场景无需「请相关方确认下游任务运行状态」提示，因为系统已确认任务实例完成。

> **实现说明**：恢复通知由告警状态变更为 `resolved` 时触发，在后端处理逻辑中追加一步群消息推送。人工恢复由用户点击「已解决」触发；自动恢复由系统在下次通知周期检测到任务实例已完成时触发。无需额外的 WebSocket/Webhook 事件订阅机制。

## 七、 页面与交互结构建议

### 7.1 入口设计 (Entry Points)
为了让数据开发工程师以最短路径触达评估功能，采用“双入口”设计：
*   **入口一：告警列表页 (List Page)**：在**每条告警卡片页脚右侧**（与「事件 ID / 负责人 / 操作人」同卡片的操作区）展示 `[影响评估]` 按钮。适用于在列表中快速选中事件、直接打开战情室评估下游影响或拉群的场景。
*   **入口二：告警详情抽屉 (Detail Drawer)**：在抽屉**头部右侧**（关闭按钮旁）展示醒目的 `[影响评估]` 主按钮。适用于查看完报错日志与进度时间线后，顺手评估影响面的场景。

#### 7.1.1 告警列表页：何时展示「影响评估」

**与告警来源无关**：数据集成、数据质量（DQC）、数据开发三类来源的展示规则**一致**。

**仅与告警状态有关**：列表卡片上的「影响评估」仅在事件仍处于**需要处置或处置进行中**的状态时出现；**已闭环**的事件在列表中不再展示该按钮，避免与「已解决」「误报」等终态抢焦点、降低误点。

| 告警状态（产品文案） | 状态值 (code) | 列表页是否展示「影响评估」 |
|---------------------|---------------|---------------------------|
| 触发中 | `firing` | **展示** |
| 已认领 / 处理中 | `acked` | **展示** |
| 已屏蔽 | `silenced` | **展示** |
| 已解决 | `resolved` | **不展示** |
| 误报 | `falsePositive` | **不展示** |

**产品说明（可写进验收）**：
* **已解决**：故障已恢复，列表主路径以复盘与归档为主；若仍需回看当时影响面，可通过运维后台/报表等后续能力扩展（V1.0 不强制在列表保留入口）。
* **误报**：事件已定性为误报，列表不再引导进入「影响评估」战情室；若需结合历史上下文复盘，可通过**告警详情抽屉**进入（见下节）。

#### 7.1.2 告警详情抽屉：「影响评估」展示规则（与列表差异）

详情抽屉头部的影响评估入口与列表页**不完全相同**，按告警状态区分为"活跃评估"和"历史记录查看"两种模式：

| 告警状态 | 详情抽屉入口 | 入口文案 | 说明 |
|---------|-------------|---------|------|
| `firing` / `acked` / `silenced` | **展示** | `[影响评估]` | 进入活跃战情室，展示实时数据 |
| `resolved`（已解决） | **展示** | `[查看影响评估]` | 进入影响评估视图（纯实时模式，隐藏操作项） |
| `falsePositive`（误报） | **展示** | `[查看影响评估]` | 进入影响评估视图（纯实时模式，隐藏操作项） |

> **与列表页的差异**：列表页在 `resolved` 和 `falsePositive` 状态下不展示任何入口（降噪），详情抽屉保留入口以支持事后复盘。复盘时采用**纯实时视角**，即展示当前最新的物理运行状态，但保留告警触发时打上的业务风险标识（如 SLA 破线、阻断等），同时隐藏底部应急操作栏。

### 7.2 核心页面：异常评估工作台 (War Room 沉浸式视图)
由于影响评估需要极大的横向和纵向屏幕空间来渲染 DAG 拓扑图，**不建议在原有的窄抽屉内直接渲染**。
点击入口后，系统弹出一个**覆盖全屏 90% 宽度的沉浸式抽屉/弹窗（战情室模式）**，采用左右分栏布局。

**页面级操作**：战情室**整体页面右上角**放置【刷新】按钮，点击后重新拉取已展开节点的实际运行状态及统计数据，手动更新画布与右侧面板。战情室打开期间不自动轮询，所有数据刷新由用户主动触发。

#### 左侧 70% 空间（上下分区）

*   **上方：当前告警信息卡片**
    *   展示当前评估主体的核心信息，让用户进入战情室后第一时间建立上下文：
    *   **事件 ID**：当前告警事件唯一标识（如 `INT-1001`）。
    *   **告警等级**：Error / Warn，以标签色区分。
    *   **告警来源**：数据集成 / 数据质量 / 数据开发。
    *   **监控事件**：具体的告警事件类型（如：离线任务失败 / 离线任务超时 / SLA 启动超时 / SLA 完成超时 / DQC 运行失败 / DQC 触发告警）。
    *   **调度周期**：任务的调度周期（如：日、周、月）。
    *   **调度批次**：任务的调度批次时间，格式为 `yyyy-mm-dd hh:mm:ss`。
    *   **任务名**：等宽字体展示完整任务名（遵循全局展示规范）。
    *   **当前告警状态**：复用现有告警状态流转定义（触发中 firing / 已认领 acked / 已屏蔽 silenced / 已解决 resolved / 误报 falsePositive）。
    *   **负责人**：格式为 `姓名(邮箱前缀)`（如 `马殿林(madianlin)`）。
    *   **触发时间**：告警首次触发的时间（如 `2026-04-02 08:23:15`）。

*   **下方：DAG 可视化画布区 (Canvas)**
    *   承载 DAG 拓扑图，提供缩放、平移、节点折叠/展开功能。
    *   顶部提供工具栏：包含全局筛选控制（影响层级、仅当前空间、仅看核心链路）与画布操作（缩放、适应视图、图例）。
        *   **影响层级**：支持选择 `1 层`、`2 层`、`3 层`，默认 `1 层`。控制后端推演和前端展示的拓扑深度。
        *   **仅当前空间**：开关控制，默认关闭（展示全部）。开启后仅展示当前项目空间内的受影响任务，过滤掉跨项目任务。
        *   **仅看核心链路**：默认关闭，画布展示用户当前手动展开的局部子图；开启后，系统基于完整潜在拓扑，自动穿透并仅展示从根节点到所有“核心任务”的关键路径，隐藏非路径节点，此时冻结节点的折叠/展开操作。
    *   图例与现有调度系统保持一致，采用以下 5 种状态色：
        *   🟢 执行成功（绿色）
        *   🔴 失败/停止（红色）
        *   🟠 运行/排队（橙色）
        *   🔵 其他状态（蓝色）
        *   ⚪ 未生成（灰色）

#### 右侧 30% 空间（上下分区）

*   **上方：AI 智能分析总结卡片**
    *   基于当前告警的影响面数据，由后端 LLM 生成一段结构化的分析总结，内容涵盖：
        *   **影响面判断**：受影响下游实例数、核心实例数、波及范围描述。
        *   **影响结论**：当前异常导致的实际业务影响（如「财务日报当前处于依赖等待状态，尚未产出」），基于实际状态描述，不做时间预测。
        *   **建议操作**：基于影响面给出的下一步操作建议（如「建议立即联系相关负责人协同处理」）。
    *   数据来源：后端 `ImpactSummary` 接口的 `aiAnalysis` 字段。
    *   **AI 分析生成策略**：
        *   **触发时机**：告警产生时即异步触发 AI 预计算，结果落库持久化。
        *   **去重机制**：以 `alertId` 为维度，同一告警同时只有一个 AI 分析作业在执行，其他请求等待同一结果。
        *   **缓存策略**：AI 分析结果一旦生成即永久生效，V1.0 不支持重新生成。打开战情室时优先读取已有结果。
    *   **AI 总结加载与降级策略**：
        *  打开影响评估时查询 AI 分析结果。若已预计算完成，直接展示；若仍在计算中，卡片显示「AI 分析生成中...」。
        *  AI 模块的不可用**不阻塞**其他数据的正常展示（统计、SLA、影响清单均独立加载）。

*   **下方：Tab 区域**
    *   **Tab 1 - 统计评估 (Stats)**：展示资产与人员统计（详见 6.5 节）。
    *   **Tab 2 - SLA 监控状态 (SLA Monitor)**：SLA 监控状态卡片列表（详见 6.6 节，展示所有配置 SLA 的任务，含破线与可控状态）。支持三个维度的筛选：**任务名称搜索**（子串匹配）、**负责人多选过滤**、**SLA状态多选过滤**（双重破线、启动破线、完成破线、可控）。
    *   **Tab 3 - 全局影响清单 (Global Impact List)**：所有告警（DQC、开发、集成）均统一展示受影响的下游**任务实例**列表（列含实例 ID、调度批次、关联任务、状态等，以后端为准）。全局清单展示根节点以下**全部**受影响下游（不依赖 DAG 当前展开状态），**数据在打开影响评估时即一次性返回**（独立于画布逐层加载的数据源），支持分页（**每页 100 条**）、**任务名搜索**（输入关键词对任务名做子串匹配，不区分大小写）以及**任务状态多选过滤、负责人多选过滤**；匹配结果卡片上仍展示该任务**当前影响/运行状态** Tag，与画布节点色系一致。切换告警或刷新清单数据源时，搜索框、状态与负责人过滤恢复默认。
    *   **Tab 4 - 日志详情 (Error Log)**：默认展示当前告警节点的报错日志摘要（与告警详情抽屉中的日志内容一致，数据源复用同一接口）。**用户点击画布上任意节点时，自动跳转到 Tab 4 并展示该节点的日志内容**，帮助用户在影响评估视图中直接查看任意节点的报错上下文，无需返回详情抽屉。DQC 组告警场景下，日志内容包含后端拼接的所有触发规则明细。**对于未运行（未生成）或依赖等待状态的节点，因不存在运行日志，Tab 4 展示空白。**

*   **底部固定操作条 (Action Bar)**：放置【一键拉群】、【一键挂起】、【级联重跑】、【一键置成功】等操作按钮（可用性详见 6.7 节按钮矩阵）。

#### 右侧面板可折叠交互

当 DAG 拓扑图规模较大（层级深、节点多）时，用户需要更多横向空间查看完整链路。右侧面板支持**折叠/收起**操作：
*   **默认状态**：左 70% + 右 30% 分栏布局。
*   **折叠状态**：点击右侧面板顶部的「◀ 收起」按钮后，面板折叠为一条窄图标条（约 40px），DAG 画布自动扩展至接近全屏宽度，画布执行 `fitView` 自适应。
*   **展开恢复**：点击图标条上的「▶ 展开」按钮，恢复原始分栏布局。

处理完故障后关闭该全屏抽屉，依然回到原列表页，不打断原有工作流。

### 7.3 终态复盘视角 (Terminal State Review)

当告警事件状态流转为 `resolved`（已解决）或 `falsePositive`（误报）时，我们称之为“终态”。

在终态下，用户仍可通过详情抽屉进入影响评估视图进行事后复盘。此时系统采用**纯实时视角**：

*   **实时物理状态**：无论是画布拓扑还是右侧统计，均实时拉取当前（打开页面时）最新的任务运行状态。即使用户在第二天打开复盘，看到的也是任务最终跑完的绿色状态。
*   **持久化风险标识**：这是复盘的核心。即使节点变绿，只要该节点曾经因为本次告警触发了业务风险（如 `SLA完成破线`、`阻断`、`可能污染`），这些红色的风险 Tag 依然会持久化展示在节点卡片和右侧列表中。
*   **交互限制**：
    *   顶部会展示一行提示：“当前告警已处于终态（已解决/误报），正在为您实时展示当前最新的拓扑与运行状态。操作栏已禁用。”
    *   底部操作栏（如“一键拉群”、“阻断下游”）将被完全隐藏/禁用，因为事件已结束，不再需要应急操作。
    *   画布的折叠/展开、缩放、刷新、全局筛选控制（层级、仅当前空间、核心链路）均保持可用，方便用户自由探索。

### 7.4 实时状态模式与风险标识解耦 (Real-time Status & Risk Decoupling)

为平衡“救火看板的实操性”与“告警原因的追溯性”，影响评估战情室（活跃状态下）采用**实时状态模式**，并通过**状态与标识解耦**的机制解决用户困惑。

*   **核心定义**：影响评估战情室是一个“实时救火看板”。画布上的节点颜色及右侧统计数据，永远反映用户打开页面（或点击刷新）那一刻的真实运行状态。
*   **状态与标识解耦**：
    *   **节点颜色（当前状态）**：只认调度系统当前的真实状态（如成功显示绿色、失败显示红色、等待显示蓝色等）。
    *   **风险标识（历史/潜在风险）**：如 `SLA完成破线`、`阻断`、`可能污染`。这些是基于告警事件上下文计算得出的属性。即使节点变绿（已被人工或系统恢复），只要该告警事件未解除，这些风险标识依然保留在节点卡片上。
*   **用户困惑消除**：为避免用户产生“节点变绿但仍在告警中”的困惑，DAG 画布左上角常驻提示文案：`“当前为实时运行状态。节点恢复后显示为成功（绿色），但保留告警触发时的风险标识。”`

## 八、 关键指标与验收标准 (Success Metrics)

1.  **故障响应效率**：从发现异常到通知所有下游负责人的时间，从平均的 N 分钟缩短至 **1 分钟内**（通过一键拉群）。
2.  **协同响应效率**：从点击「一键拉群」到应急群创建成功并推送报告卡片的时间不超过 **30 秒**。
3.  **系统性能**：拓扑图逐层展开单次渲染响应时间不超过 **2 秒**。

## 附录：Agent 调用影响评估的场景还原

为便于理解影响评估作为“中立服务”如何被 Agent 调用（即 PRD 2.2 节所述场景），特补充以下对话还原：

**场景假设**：用户在群里向「DataOps 运维助手 (Agent)」提问。

> **用户**：“@运维助手，`dwd_user_order` 这个任务今天失败了，帮我看看怎么回事？”

**Agent 的处理逻辑**：
1. **第一步（查日志找原因）**：Agent 调用日志接口，发现：“是因为上游的一张表缺字段报错了。”
2. **第二步（查影响面做判断）**：Agent 需要知道这个任务失败的后果严不严重，需不需要紧急叫人。此时，Agent **在后台静默调用「影响评估服务」**。
   *   **Agent 发送请求**：`Input: Target=dwd_user_order, Context=Agent`
   *   **影响评估服务返回**：`Output: 影响了下游 50 个任务，其中包含 3 个核心 SLA 任务（财务日报、CEO看板），牵连了张三、李四等 5 个负责人。`

**拿到上述 Output 后，Agent 才算真正“获取了上下文”，从而给出聪明的、有针对性的回答：**

> **Agent 最终回复用户**：“`dwd_user_order` 失败是因为上游缺字段。**注意！这个失败已经导致下游的『CEO看板』和『财务日报』等 3 个核心任务处于等待状态，预计影响张三、李四等 5 人。** 建议您立即在战情室一键拉群处理！”
